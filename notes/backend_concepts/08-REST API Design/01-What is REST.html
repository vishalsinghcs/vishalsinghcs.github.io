What is REST?

REST stands for Representational State Transfer — an architectural style for building networked applications, defined by constraints that make systems scalable, interoperable, and predictable.

The Origin: Why REST Exists

In 1990, Tim Berners-Lee invented the World Wide Web along with the first versions of URI, HTTP, and HTML. The system was small and manageable at first, but the web grew faster than anyone anticipated. By 1993, the web was heading toward collapse under the weight of its own exponential user growth — the original infrastructure had simply not been designed for this scale.

Roy Fielding, co-founder of the Apache HTTP Server project, stepped in. To make the web more scalable, he proposed a set of architectural constraints. He then worked alongside Berners-Lee to co-author the HTTP/1.1 specification in 1997. In the year 2000, Fielding formalized these ideas in his PhD dissertation and gave them a name: Representational State Transfer (REST). What we call "REST APIs" today is rooted in that dissertation.

The Six REST Constraints

Fielding defined six constraints that, taken together, describe the REST architectural style:

  1. Client-Server: The client handles the UI and user experience; the server handles data storage and business logic. They are separated and can evolve independently. This is the foundation of modern frontend/backend architecture.
  2. Uniform Interface: A standardized, consistent contract governs how components communicate. This includes resource identification (via URIs), manipulation through representations, self-descriptive messages, and HATEOAS (Hypermedia as the Engine of Application State). Uniformity reduces guesswork for API consumers.
  3. Layered System: The architecture is hierarchical — each layer can only see and interact with its immediate neighbour. This allows for load balancers, proxies, caches, and security layers to be added or changed without affecting the core system.
  4. Caching: Server responses must be explicitly labeled as cacheable or non-cacheable. When clients cache valid responses, server load drops, latency decreases, and user experience improves.
  5. Stateless: Every request from a client to the server must contain all information needed to process it. The server retains no memory of previous requests. This improves reliability and scalability — any server instance can handle any request.
  6. Code on Demand (Optional): Servers can optionally send executable code (e.g., JavaScript) to extend client functionality. This is the only optional constraint and is relatively rare in modern API design.

What Does "Representational State Transfer" Mean?

The name itself is a compressed description of what REST does:

Word Meaning Example
Representational Resources can be represented in multiple formats depending on context A user object can be JSON (for APIs), HTML (for browsers), or XML (for legacy systems)
State The current condition or attributes of a particular resource A shopping cart's state = list of items, quantities, and total price
Transfer Movement of resource representations between client and server A GET request transfers the JSON representation of a user from server to client

Why Follow REST Standards?

REST is not a strict protocol — it is an architectural style. This means there is no enforcement. However, following REST standards provides major practical advantages:

  • Eliminates guesswork: When a backend follows established conventions, API consumers (frontend engineers, mobile developers, third-party integrators) can correctly predict API behavior without reading source code.
  • Fewer integration errors: Consistent, predictable APIs mean less debugging, fewer misunderstandings between teams, and fewer support calls.
  • Interoperability: REST APIs can be consumed by any HTTP-capable client — browsers, mobile apps, server-to-server calls, IoT devices.
  • Scalability: The stateless constraint means any server in a horizontally scaled cluster can handle any incoming request.

The API Design Workflow

Before writing a single line of code, a backend engineer should follow this planning workflow:

The key insight from the lecture: you design the interface before you write the code. Use tools like Insomnia, Postman, or Swagger to prototype your API endpoints — this forces you to think about the consumer's experience, not just the server implementation.

Mental Model

REST is like a postal system standard. The standard defines how addresses should look, how envelopes should be sealed, and what stamps are required. Any postal worker (server) can handle any letter (request) because the format is universal. Without the standard, every sender and recipient would have to negotiate their own format — chaos.

Important Terms

  • REST: Representational State Transfer — an architectural style, not a protocol.
  • Resource: Any identifiable thing on a server (user, product, order). Resources are the nouns of REST.
  • Representation: The format in which a resource is transmitted (JSON, HTML, XML).
  • Stateless: No session state stored on the server between requests.
  • HATEOAS: Hypermedia as the Engine of Application State — a sub-constraint of Uniform Interface.

Key Takeaways

  • REST was invented to solve the web's scalability crisis in the 1990s.
  • It is defined by 6 constraints — statelessness and uniform interface are the most impactful.
  • Following REST reduces integration errors because consumers can make safe assumptions about your API.
  • Design the API interface first, before coding. The interface is the contract.

Quick Revision

  • 6 constraints: Client-Server, Uniform Interface, Layered, Cacheable, Stateless, Code-on-Demand.
