CRUD Operations in Practice

CRUD — Create, Read, Update, Delete — maps onto HTTP methods and URL patterns in a consistent, predictable way. Once you understand the pattern for one resource, you can apply it to every resource in your system.

The Full CRUD Pattern

For any resource (let's use organizations as the example), the five standard CRUD endpoints follow this exact pattern:

Operation Method URL Body? Success Code Response
List all GET /organizations No 200 Paginated list
Create POST /organizations Yes 201 Newly created object
Get one GET /organizations/:id No 200 The resource object
Update PATCH /organizations/:id Yes (partial) 200 Updated resource object
Delete DELETE /organizations/:id No 204 Empty (no content)

Key pattern: List and Create share the same URL (/organizations) — differentiated only by HTTP method. Get-one, Update, and Delete all share the same URL (/organizations/:id) — also differentiated only by HTTP method. The server uses the method as the differentiator, not a different URL.

Create — POST

The client sends a JSON body with only the fields the server requires. Server-managed fields (id, createdAt, updatedAt) must never be accepted from the client — these are always generated on the server side.

  • Client sends: {"name": "Acme Corp", "status": "active", "description": "Our org"}
  • Server responds: 201 Created with the full newly created object including the server-generated id and createdAt.

List — GET

Returns a collection of resources. For a well-designed list API, the response always includes pagination metadata (covered in the next note). The status code is always 200, even if the list is empty.

  • Never return 404 on a list API just because there is no data. An empty list is a valid result — return [] with a 200 status. 404 is for when a specific requested resource doesn't exist.

Get One — GET with ID

Fetches a single resource by ID. The ID is passed as a dynamic parameter in the URL path (/organizations/abc123), never in the request body.

  • If found: Returns 200 with the resource object.
  • If not found: Returns 404 Not Found. This is the correct use case for 404 — a specific resource was requested but does not exist.

Update — PATCH with ID

Sends only the fields that need to change. The server merges the partial payload with the existing resource.

  • Client sends: {"status": "active"} — only the field being updated.
  • Server responds: 200 OK with the full updated resource object.

Delete — DELETE with ID

No request body is needed. The ID in the URL is sufficient to identify the resource to delete.

  • Server responds: 204 No Content — the operation was successful (200-series) but there is no data to return because the resource was just deleted.
  • Response body: Empty. Nothing to serialize back.

HTTP Status Code Reference

Code Name When to Use
200 OK Successful GET, PATCH, or custom action response
201 Created Successful POST that created a new resource
204 No Content Successful DELETE — no response body
400 Bad Request Validation error — client sent invalid data
401 Unauthorized Missing or invalid authentication token
403 Forbidden Authenticated but not authorized for this resource
404 Not Found Specific resource requested by ID does not exist
409 Conflict Duplicate resource (e.g., unique constraint violation)
429 Too Many Requests Rate limit exceeded
500 Internal Server Error Unhandled server-side error

The Pattern Is Reusable

This is the key insight from the lecture. Once you've designed CRUD for organizations, you apply the exact same pattern to projects, tasks, and every other resource. The URL structure changes (the resource name changes), but the method-to-operation mapping, the request body rules, and the status codes remain identical. Consistency is the goal — and this pattern is the tool.

Mental Model

The URL is like a noun and the HTTP method is the verb. /organizations (noun). GET it (verb), POST to it (verb), DELETE from it (verb). Combining the same noun with different verbs produces completely different operations — without any ambiguity.

Important Terms

  • 201 Created: Returned when a POST creates a new resource.
  • 204 No Content: Returned by DELETE — success but no body to send.
  • 404 Not Found: Only for specific resource lookups — never for empty lists.
  • Dynamic Parameter (:id): Variable in the URL representing a specific resource instance.
  • Server-Managed Fields: id, createdAt, updatedAt — generated by the server, never accepted from client.

Key Takeaways

  • List + Create share the same URL. Get-one + Update + Delete share the same URL. Method is the differentiator.
  • POST → 201. PATCH → 200. DELETE → 204. GET → 200.
  • An empty list returns 200 (not 404). 404 is only for specific resource lookups by ID.
  • The CRUD pattern is the same for every resource — only the resource name changes.

Quick Revision

  • GET /resources → 200, list (with pagination)
  • POST /resources → 201, returns new object
  • GET /resources/:id → 200 or 404
  • PATCH /resources/:id → 200, returns updated object
  • DELETE /resources/:id → 204, empty body
  • Empty list: Return [] with 200, not 404
