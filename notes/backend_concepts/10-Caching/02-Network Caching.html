Network Caching

As a backend engineer, you will deal with caching at three major levels: Network, Hardware, and Software (In-Memory). The Network level consists heavily of Content Delivery Networks (CDNs) and Domain Name System (DNS) caching.

1. Content Delivery Networks (CDN)

The goal of a CDN is to cache content on servers that are geographically closer to the end users. This minimizes latency and reduces the computational and network load on the originating server.

How a CDN Works

  1. User Request: A user requests a resource (e.g., an image, video, or web page) via a URL.
  2. DNS Resolution: The browser sends a DNS query. The CDN's DNS system routes the request to the nearest PoP (Point of Presence). A PoP is a concentrated region containing multiple Edge Servers.
  3. Routing Decisions: The request is routed based on the user's geographic location and network conditions. (e.g., A user with a poor connection might be routed to a server containing a lower-quality video format).
  4. Edge Server Check:
    • Cache Hit: If the content is on the edge server, it is returned instantly.
    • Cache Miss: If not, the edge server fetches the resource from the originating server (e.g., servers in the US), serves it to the user, and caches a copy for future requests.

CDNs utilize TTL (Time to Live) to decide how long a piece of content stays in the cache. Once the TTL expires, the next request will fetch a fresh copy from the originating server and reset the TTL.

2. DNS Caching

DNS queries convert domain names (example.com) into IP addresses. Because resolving a domain from scratch involves querying multiple servers globally (Root, Top-Level Domain, Authoritative), DNS heavily relies on caching in multiple layers to avoid this slow recursive process for every request.

The Hierarchy of DNS Caching

When you type a domain and press enter, caching is checked in this exact order:

Layer Component Description
1 Operating System Cache Windows, Mac, and Linux maintain their own local DNS cache. This is the absolute first place checked.
2 Browser Cache Modern browsers (Chrome, Firefox) maintain their own internal DNS cache separate from the OS.
3 Recursive Resolver Cache Provided by your ISP (e.g., Airtel, Jio, Comcast) or a public DNS provider (e.g., Google 8.8.8.8, Cloudflare 1.1.1.1). They cache results for thousands of users in your region.
4 Authoritative Name Server Cache The final destination that holds the actual IP. Even these servers sometimes implement caching to skip internal lookups.

If a Cache Hit occurs at any of these layers, the IP is returned immediately, and the remaining layers are skipped entirely.

Mental Model

Think of a CDN like a global chain of fast-food franchises. The original recipes and ingredients come from the headquarters (originating server), but the actual food is prepared and served at a branch right down your street (Edge Node).

Think of DNS Caching like saving a friend's phone number. You look it up in the phonebook once (Recursive Resolution), but after that, you save it in your phone contacts (Local Cache) so you never have to look it up again.

Important Terms

  • Edge Nodes/Servers: Servers placed geographically close to end users.
  • PoP (Point of Presence): A specific region or data center containing multiple edge servers.
  • TTL (Time to Live): The expiration timer placed on cached data.
  • Recursive Resolver: The DNS server (usually your ISP) that hunts down an IP address if it isn't in your local cache.

Key Takeaways

  • Network caching happens mostly invisibly to developers via CDNs and DNS.
  • CDNs route users to the nearest PoP based on location and network strength.
  • DNS relies on a 4-tier caching hierarchy to prevent the internet from collapsing under the weight of DNS queries.

Quick Revision

  • CDN: Edge nodes, PoP, Geographic proximity.
  • TTL: Defines how long content lives in the CDN before fetching fresh from origin.
  • DNS Cache Hierarchy: OS → Browser → ISP/Resolver → Authoritative.
