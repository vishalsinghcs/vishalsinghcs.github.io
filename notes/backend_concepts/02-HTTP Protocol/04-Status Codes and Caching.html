Status Codes and Caching

HTTP response codes universally communicate the result of a request. HTTP caching uses headers to store copies of responses, reducing server load and bandwidth.

Response Status Codes

Status codes standardize communication across all web services, meaning clients can handle outcomes programmatically without parsing body content.

1xx: Informational

  • 100 Continue: Server received headers and client can proceed to send the body (used in large uploads).

2xx: Success

  • 200 OK: Request successful (common for GET requests).
  • 201 Created: Request fulfilled and a new resource was created (common for POST requests).
  • 204 No Content: Request successful but no body is returned (often used for DELETE or OPTIONS pre-flight).

3xx: Redirection

  • 301 Moved Permanently: The resource has moved permanently to a new URL. Future requests should use the new URL.
  • 302 Found (Temporary Redirect): The resource has moved temporarily. Clients should keep using the original URL.
  • 304 Not Modified: The resource has not changed since the last request. The client should use its cached copy.

4xx: Client Errors

These occur when the client sends invalid, illogical, or unauthorized data.

  • 400 Bad Request: Invalid request format or illogical data (e.g., sending a string when a number was expected).
  • 401 Unauthorized: Missing, expired, or invalid authentication credentials (e.g., JWT token).
  • 403 Forbidden: The server understands the request, but refuses to authorize it (e.g., trying to access another user's data).
  • 404 Not Found: The requested resource does not exist.
  • 405 Method Not Allowed: Using the wrong HTTP method (e.g., trying to PUT to a route that only accepts POST).
  • 409 Conflict: The request conflicts with current server state (e.g., creating a folder with a name that already exists).
  • 429 Too Many Requests: Client has been rate-limited.

5xx: Server Errors

These occur when something goes wrong on the server's end.

  • 500 Internal Server Error: An unexpected condition or unhandled exception broke the process.
  • 501 Not Implemented: The server does not support the requested functionality (yet).
  • 502 Bad Gateway: Typically returned by reverse proxies (like Nginx) when the upstream server returns an invalid response.
  • 503 Service Unavailable: The server is down (e.g., undergoing maintenance or handling too much traffic).
  • 504 Gateway Timeout: The upstream server failed to respond within the allowed time.

HTTP Caching

Caching stores a copy of a given response so that subsequent requests can use the stored copy, bypassing the need to fetch data from the server again.

How it Works

  1. Initial Request: The server responds with the payload alongside three important headers:
    • Cache-Control: Defines caching policies (e.g., max-age=10).
    • ETag: A unique hash/identifier of the response object.
    • Last-Modified: The timestamp when the resource was last altered.
  2. Subsequent Request: The client sends a new request asking if the resource is still valid using:
    • If-None-Match: Sends the ETag to verify.
    • If-Modified-Since: Sends the Last-Modified timestamp.
  3. Server Validation: If the hash matches or the timestamp hasn't changed, the server responds with a 304 Not Modified and no body. The client reuses its cached copy.

Note: While native HTTP caching is powerful, modern applications often rely on client-side libraries (like React Query) for more granular, programmable cache invalidation.

Key Takeaways

  • 2xx = Success
  • 3xx = Go elsewhere / Use cache
  • 4xx = You (the client) messed up
  • 5xx = We (the server) messed up

Quick Revision

  • Use 201 when POSTing new data successfully.
  • Use 401 for missing logins, 403 for lack of permissions.
  • Use 400 for validation errors.
  • ETag acts as a unique hash. If the hash hasn't changed on the server, a 304 is returned.
