Interview Questions

11 - Background Tasks and Task Queues: Workflows, Task Types, Design Considerations, and Best Practices.

These questions test your ability to reason about asynchronous architectures, worker management, data consistency, and edge-case failures in a high-traffic production system. The goal is engineering thinking, not recitation of terms.

1. Scenario-based Questions

  • A background worker processing email sending tasks crashes midway because the 3rd-party email API goes down. You are using BullMQ with an exponential backoff strategy, but after 5 retries, the task is marked as failed. How would you handle this to ensure the email is eventually sent when the 3rd-party API recovers a day later?
  • You implement a chained task workflow for video uploading: (1) Encode Video, (2) Generate Thumbnails, (3) Generate Subtitles. Users complain that videos take too long to appear. Upon investigation, you realize that if Thumbnail generation fails, the entire chain is retried from the beginning, re-encoding the 2GB video. How would you redesign this workflow?
  • Your application creates a "Delete Account" batch task that spans multiple microservices. Midway through the deletion process, the database is locked for maintenance and the task crashes. The next time it runs, it crashes with a "foreign key constraint" or "user not found" error because half the data was already deleted. What architectural principle was violated, and how do you fix it?
  • A spike in user sign-ups adds 10,000 "Send Welcome Email" tasks to your Redis broker. Your 2 consumer nodes are processing 10 tasks per second. Users are receiving their welcome emails 20 minutes late, but your CPU usage on the workers is only at 10%. Why is processing so slow, and what is the immediate fix to handle this backlog?
  • You deploy a new version of your worker codebase that expects a new JSON field user_preferences in the task payload. Suddenly, thousands of tasks start failing immediately. What deployment mistake did you make regarding background tasks, and how do you safely roll out schema changes to task payloads?
  • You have a recurring cron task that cleans up orphan database sessions every midnight. However, because you are running 3 instances of your backend API behind a load balancer, the cleanup task executes 3 times simultaneously, causing database deadlock errors. How do you ensure a cron job runs exactly once across a distributed system?
  • Your external PDF generation service is strict about rate limits: 100 requests per minute. You have 5 worker nodes processing report generation tasks. How do you implement a distributed rate limiter to ensure the combined workers never exceed 100 requests per minute?
  • You notice that your Redis instance memory is growing uncontrollably, reaching 100% and crashing the application. Upon inspection, the task queue is not particularly large, but the worker nodes are logging a high number of "Timeout" errors when acknowledging tasks. Explain the interaction between visibility timeouts, unacknowledged tasks, and memory leaks in the broker.

2. "What if?" Questions

  • What if a broker (like RabbitMQ) did not enforce a visibility timeout, and instead immediately deleted a task the moment a consumer picked it up? How would this impact the reliability of payment processing pipelines?
  • What if you designed a distributed system where background tasks were processed completely asynchronously, but the frontend required the immediate result of the task to display a success message? How would you bridge the asynchronous gap to make the frontend feel synchronous?
  • What if you used a standard relational database (like PostgreSQL) table as a task queue instead of a dedicated broker like SQS or Redis? What specific concurrency and performance bottlenecks would you face under high throughput?
  • What if your background tasks were perfectly idempotent, but the downstream 3rd-party API they call (e.g. charging a credit card) is not idempotent? How do you safely retry your background tasks without double-charging the user?

3. Debugging Questions

  • You have a queue dedicated to processing image uploads. The queue length is constantly sitting at zero, but users are complaining that their images are never processed. You check the broker, and tasks are being created successfully. Where could the disconnect be happening between the producer, broker, and consumer?
  • A user reports that every time they click "Export Data", they receive 5 duplicate emails containing the same CSV file. You check the codebase, and the producer definitely only enqueues the task once. Explain why a worker might process and complete the exact same task 5 times.
  • Your worker nodes are suddenly restarting every 15 minutes due to Out Of Memory (OOM) errors. The tasks they are processing are simple data aggregations. You suspect the tasks themselves are fine, but the worker process is holding onto memory. What is a common pitfall in worker process lifecycle management that leads to memory leaks?

4. Design Questions

  • Design a background task architecture for a food delivery app like UberEats. Specifically, handle the workflow where an order is placed, the restaurant accepts it, and a delivery driver is assigned. Outline the queues, the task types, and the error handling required for this flow.
  • Design a notification system that sends a monthly newsletter to 10 million users. You must ensure that no user receives the email twice, the database isn't locked during processing, and the entire job completes within 4 hours.
  • Design a system to handle webhooks from Stripe (e.g. payment_intent.succeeded). Stripe expects a 200 OK within 3 seconds, or they will retry. How do you design your API and background workers to ingest, validate, and process these webhooks reliably at scale?

5. Architecture & Trade-off Questions

  • Discuss the trade-offs between using a Push-based queue (where the broker actively sends tasks to workers) versus a Pull-based queue (where workers continuously poll the broker for tasks). When would you choose one over the other?
  • Why would an engineering team choose to deploy background workers as completely separate isolated services (e.g., in a separate Kubernetes deployment) rather than running the worker threads in the same Node.js/Python process as the main HTTP API? What are the operational costs of this isolation?
