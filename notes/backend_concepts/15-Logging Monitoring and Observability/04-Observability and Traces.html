Observability and Traces

Connecting the dots to understand exactly where a request failed.

We call a backend system observable when we can determine its internal state purely by looking at external outputs. Observability is a relatively modern practice that relies on three core pillars coming together.

The Three Pillars of Observability

  • Logs: Tell us what happened.
  • Metrics: Tell us about patterns and trends.
  • Traces: Tell us about component interactions.

What are Traces?

You can imagine traces as transactions. When a request hits your infrastructure (whether originating from a front-end client, a load balancer, or another backend), a trace tracks that specific request as it travels through your system.

A single trace will show you exactly how the request touched the Handler Layer → Validation Layer → Service Layer → Repository Layer → Database Layer.

The Debugging Workflow

When you have logs, metrics, and traces properly configured, debugging an incident in production becomes a highly streamlined workflow:

1. Alert triggers (Slack: Error rate > 80%)
↓
2. View Metric (See high failure count on Dashboard)
↓
3. View Logs (Find specific 500 errors associated with the metric)
↓
4. View Trace (Click a log to see exactly which function/DB call failed)

Instead of guessing which microservice or function broke, the trace visually points you to the exact line or database query that failed in the transaction, instantly isolating the bug.

Mental Model

Think of a Trace like a shipping tracker for a package. It shows you exactly when it left the warehouse (handler), hit the sorting facility (service logic), and failed delivery at the destination (database).

Important Terms

Trace: The entire lifecycle of a request as it traverses through various services and layers in a distributed architecture.

Span: A single unit of work within a trace (e.g., a specific database query or API call).

Key Takeaways

  • Metrics tell you there's a fire.
  • Logs tell you what caught fire.
  • Traces show you the path the fire traveled.

Quick Revision

  • Three Pillars: Logs, Metrics, Traces.
  • Workflow: Alert → Metric → Log → Trace.
  • Goal: Find exact component failure instantly.
