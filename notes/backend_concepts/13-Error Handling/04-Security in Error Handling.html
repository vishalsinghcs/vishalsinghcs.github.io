Security in Error Handling

Error messages are often leveraged by malicious actors to map out the internal structure of your application. Securing your error handling workflow is paramount to protecting both your platform and your users.

Leaking Internal Details

Never return raw database or infrastructure errors directly to the client. A naive error response might contain table names, column names, or stack traces. Malicious users can use this information to construct highly targeted SQL Injection attacks.

The Fix: Catch these errors in your global middleware and abstract them. If an error reaches your default "catch-all" block, always return a generic 500 Internal Server Error with a message like "Something went wrong," keeping internal state hidden.

Authentication endpoints (like login) are prime targets. Consider a scenario where a user submits an email and password:

  • If you return "User with this email does not exist", an attacker can enumerate your database to figure out which emails belong to registered users.
  • If you return "Incorrect password", the attacker knows the email is valid and can focus solely on brute-forcing the password.

The Fix: Regardless of whether the email exists or the password was wrong, always return a generic, indistinguishable error: "Invalid username or password." This prevents account enumeration attacks.

Log Redaction

Errors must be logged for debugging, but logs are often shipped to external visualization tools (like ELK Stack or Datadog). In major data breaches, hackers often target these log aggregation services.

Never log sensitive information such as:

  • Passwords or API Keys
  • Credit card numbers (PII)
  • User email addresses

The Fix: Log the user's UUID and a generated Correlation ID instead. This gives engineers enough context to trace the error through the system without exposing personally identifiable information.

Mental Model

Treat error messages like poker tells. If you give away too much specific detail, the opponent (attacker) can easily deduce your hand (internal architecture).

Important Terms

Account Enumeration: An attack where a malicious user probes login endpoints to discover which emails are registered on the platform.

Log Redaction: The process of removing or masking sensitive data before it is written to log files.

Key Takeaways

  • Never expose database structure, column names, or stack traces in API responses.
  • Always return a generic "Invalid username or password" for login failures.

Quick Revision

  • 500 Errors = Hide details.
  • Auth Errors = Generic messages.
  • Logs = Redact PII, log UUIDs.
