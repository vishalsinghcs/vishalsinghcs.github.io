Hardware & In-Memory Caching

While network caching handles data in transit, software-based caching (what backend engineers write code for) relies entirely on the architecture of Hardware memory.

Hardware Level Caching

Before understanding software caches, you must understand how a computer's CPU and memory interact. Computers have multiple layers of storage, trading off capacity for speed:

  1. CPU Caches (L1, L2, L3): Tiny amounts of extremely fast memory built directly onto the CPU chip. CPUs use predictive algorithms to move data here right before it's needed (e.g., sequentially reading an array in memory is fast because the CPU caches the next blocks ahead of time).
  2. Main Memory (RAM): Also known as Random Access Memory. Very fast, but limited in capacity.
  3. Secondary Storage (Hard Disks/SSD): Massive capacity, but mechanically or electronically slower. Provides permanent data persistence.

RAM vs. Disk (The Core Trade-off)

Feature RAM (Primary Storage) Disk (Secondary Storage)
Speed Extremely Fast (Electrical signals, random access O(1)) Slower (Mechanical parts in HDDs, block-level access)
Capacity Low (Measured in GBs) High (Measured in TBs)
Volatility Volatile (Data is lost when power is turned off) Persistent (Data survives reboots)

In-Memory Key-Value NoSQL Databases

Because RAM is so fast, we use specific database technologies designed explicitly to store data in RAM rather than on Disk. The most popular examples are Redis, Memcached, and cloud equivalents like AWS ElastiCache.

Let's break down the name: In-Memory Key-Value NoSQL Database

  • In-Memory: Data is stored in RAM (Primary Storage) for ultra-fast retrieval, unlike Postgres or MySQL which default to Disk.
  • Key-Value: The data model is simple. You provide a unique "Key", and the database stores a "Value" (which can be a string, JSON, list, etc.).
  • NoSQL: They do not enforce strict schemas, tables, or relational constraints. They are built for raw speed, not complex relationships.

How Do They Survive Reboots? (Persistence)

If RAM is volatile, what happens when a Redis server crashes? Does all the cached data disappear?

While the primary operation (reading and writing) happens in RAM for speed, technologies like Redis implement background mechanisms to occasionally sync that data to Secondary Storage (Disk). When the server restarts, Redis reads the data from the disk back into RAM.

Note: The primary goal is speed. The disk is only used as a safety net, not for querying.

Mental Model

Think of Disk Storage as a filing cabinet in the basement. It holds everything securely, but takes time to walk down and search through.

Think of RAM (In-Memory) as the sticky notes on your monitor. You can only fit a few of them (low capacity), and if the wind blows them away they are gone (volatile), but you can read them instantly without getting out of your chair.

Important Terms

  • Volatile: Memory that loses its data when the system powers down.
  • Random Access: The ability to access any item of data in a constant amount of time (O(1)), regardless of its physical location in the memory.
  • Key-Value Store: A database paradigm resembling a giant dictionary or hash map.

Key Takeaways

  • We trade capacity and persistence for raw speed when using RAM.
  • Redis and Memcached are the industry standards for software-based, in-memory caching.
  • These tools are "NoSQL" because relational overhead (joins, constraints) defeats the purpose of raw speed.

Quick Revision

  • L1/L2/L3: CPU caching (predictive).
  • RAM: Fast, Volatile, Low Capacity.
  • Disk: Slow, Persistent, High Capacity.
  • Redis: In-Memory, Key-Value, NoSQL.
