Services and Repositories

The two layers below the controller that execute all business logic and database operations, keeping each concern cleanly isolated.

The Service Layer

After the controller validates and binds the request data, it calls the appropriate service method. The service layer is where all the actual processing of your API happens.

A key rule: looking at a service method, you should not be able to tell it is used in an HTTP server. It should read like a plain function that takes some data, does work, and returns a result — no HTTP status codes, no request/response objects.

What a Service Does

  • Orchestrates business logic: Sending emails, calling external APIs, computing results, applying business rules.
  • Calls repository methods: When data needs to be read from or written to the database, the service delegates to the repository.
  • Merges data: A single service method can call multiple repository methods and combine their results into one response.
  • No HTTP concerns: The service never decides what status code to return — that is the controller's responsibility.

The Repository Layer

The repository layer has a single, narrow responsibility: database operations. It receives data from the service, constructs the database query, executes it, and returns the raw result.

Design Rules for Repositories

  • One responsibility per method: A method to fetch all books should only fetch all books. A method to fetch one book by ID should only do that. Never use an optional parameter to change the fundamental behavior of a repository method.
  • No business logic: The repository constructs queries. It does not apply rules, send emails, or make external calls.
  • Returns raw data: It returns whatever the database gives back, and the service layer decides what to do with it.

Layer Interaction Pattern

Why This Three-Layer Split?

There is no hard requirement to use three layers — you technically could put everything in a single handler function. But as an application scales, mixing all three concerns causes serious problems:

Layer Concern If Missing
Controller HTTP I/O — parsing, validating, responding Business logic becomes untestable in isolation
Service Business logic — orchestration, rules, external calls Logic is scattered across handlers, causing duplication
Repository Database queries only Changing databases or query logic requires touching business logic

Mental Model

Service = Kitchen Chef. The chef (service) receives an order (from the controller), decides what to cook and how, calls on the pantry (repository) for ingredients (database data), and returns the prepared dish. The chef doesn't care about how the order arrived — that's the receptionist's (controller's) problem.

Important Terms

  • Service Layer: The layer that executes all business logic, orchestrating calls to repositories and external services.
  • Repository Layer: The layer that exclusively handles database query construction and execution.
  • Orchestration: A service method calling multiple repositories and combining their results.

Key Takeaways

  • Service methods should not contain any HTTP-specific code (no status codes, no req/res).
  • Repository methods have one job — one method, one type of database operation.
  • A service can use many repositories; a repository should do one thing.
  • Swapping a database only requires changing the repository, not the service or controller.

Quick Revision

  • Controller calls: Service layer.
  • Service calls: Repository (+ external APIs, emails, etc.).
  • Repository calls: Database only.
  • Service: Can use many repos; knows business rules.
  • Repository: One operation per method; no logic.
