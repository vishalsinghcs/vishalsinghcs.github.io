Writing Queries — JOINs and CRUD

With the schema in place, every API endpoint maps to one or more database queries. Understanding how to write efficient SELECT queries with JOINs, embed related data as JSON, filter dynamically, paginate results, and safely pass user-provided values is the day-to-day work of a backend engineer.

How to Approach SQL Queries

A common mistake is to write the SELECT clause first. Instead, always start from the FROM clause — this forces you to first determine where your data is coming from before deciding what fields to retrieve. 

-- Build queries in this mental order:
FROM   users u          -- 1. Where is the data?
JOIN   user_profiles up -- 2. What tables do I need to join?
WHERE  u.id = $1        -- 3. What is the filter condition?
ORDER BY u.created_at DESC -- 4. What is the sort order?
SELECT u.*, ...         -- 5. Which fields do I actually need?

Aliases

Assigning a short alias to a table (e.g., FROM users u, FROM user_profiles up) makes longer queries far more readable. You can then reference u.email instead of users.email throughout the query. It also disambiguates column names when joining tables that share column names.

JOINs — Combining Data from Multiple Tables

A JOIN merges rows from two tables based on a matching condition (usually a foreign key equals a primary key). The two most commonly used JOIN types are:

JOIN Type Behaviour When to Use
INNER JOIN Only returns rows where the condition matches in both tables When the related row is guaranteed to exist
LEFT JOIN Returns all rows from the left table even if no match exists in the right table (NULL for missing right-side columns) When the related row might not exist (optional relationship)

Practical example: When fetching users with their profiles, use LEFT JOIN because a user might not have filled in their profile yet. An INNER JOIN would silently omit users with no profile entry — a data bug that is hard to detect. 

SELECT
  u.*,
  to_jsonb(up.*) AS profile   -- Embed the profile row as JSON
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
ORDER BY u.created_at DESC;

to_jsonb(up.*) is a Postgres function that converts an entire row from the user_profiles table into a jsonb object. This allows you to return nested, related data in a single query call — no second API roundtrip needed.

Parameterized Queries — Security Against SQL Injection

Whenever a query includes a dynamic value from the user (a URL parameter, a request body field), you must use a parameterized query — never string concatenation.

A parameterized query contains a placeholder (a "slot") for the dynamic value. The database driver fills this slot at execution time, treating whatever value is provided as a plain string — not executable SQL. This prevents SQL injection attacks, where a malicious user embeds SQL commands in an input field (e.g., entering '; DROP TABLE users; -- as their name). 

-- Dangerous (SQL injection risk):
"SELECT * FROM users WHERE id = '" + userId + "'"

-- Safe (parameterized):
SELECT * FROM users WHERE id = $1   -- $1 is the placeholder
-- Pass userId as the parameter value at runtime

All database drivers (Postgres drivers for Node.js, Go, Python, etc.) support parameterized queries. Always use them when incorporating user-supplied values in a query.

READ — GET Single Record

-- GET /users/:id → get one user with their profile
SELECT
  u.*,
  to_jsonb(up.*) AS profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
WHERE u.id = $1;    -- $1 is the user ID from the URL parameter

READ — GET List with Filtering, Sorting, and Pagination

For list APIs, the query must be dynamically constructed in the application layer. The server checks which optional parameters the client sent (filter, sort, page) and builds the WHERE, ORDER BY, and LIMIT/OFFSET clauses accordingly. 

-- GET /users?letter=J&sortBy=email&sortOrder=asc&page=2&limit=10
SELECT
  u.*,
  to_jsonb(up.*) AS profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
WHERE u.full_name ILIKE $1    -- $1 = 'J%'  (case-insensitive starts-with)
ORDER BY u.email ASC          -- constructed dynamically based on sortBy / sortOrder
LIMIT $2                      -- $2 = limit (e.g., 10)
OFFSET $3;                    -- $3 = (page - 1) * limit  (e.g., page 2 → offset 10)

Key details:

  • ILIKE is case-insensitive pattern matching. The % wildcard means "any characters after this point". So 'J%' matches all names starting with J regardless of case.
  • OFFSET is zero-based. If the user passes page=2 and limit=10, the query should use OFFSET = (2 - 1) * 10 = 10. Convert from user-facing 1-indexed pages to 0-indexed offsets in the backend.
  • The ORDER BY field and direction must be validated in application code before being used in the query — never insert user-provided sort field names directly, as that could expose column names or enable injection.
  • The default sort (when client doesn't specify) should always be ORDER BY u.created_at DESC.

CREATE — INSERT

-- POST /users → create a new user
INSERT INTO users (email, full_name, password_hash)
VALUES ($1, $2, $3)
RETURNING *;   -- Return the newly created row (including server-generated id and created_at)

The RETURNING * clause makes Postgres return the inserted row immediately after the INSERT, avoiding a second SELECT query. This is how the API returns the newly created resource in the response body with its server-generated id and created_at.

UPDATE — PATCH (Partial Update)

-- PATCH /users/:id → update only the fields the client sent
UPDATE user_profiles
SET
  bio   = $1,    -- Only fields the client actually passed are included
  phone = $2
WHERE user_id = $3
RETURNING *;      -- Return the updated row

In practice, the backend first checks which fields the client provided, then dynamically constructs the SET clause to include only those fields. Fields the client did not send are not included in the query at all — they retain their existing values. This is how PATCH (partial update) differs from PUT (full replacement).

The updated_at column should also be updated on every PATCH. This can be done explicitly in the SET clause, or automatically via a database trigger (covered in the next note).

Mental Model

A parameterized query is like a form letter with blank fields. The query is the template: "Dear [Name], your order [Order ID] has shipped." The blanks are filled in later — but they are always treated as data, never as instructions. If someone writes "DELETE FROM orders" in the Name field, it's just an unusual name — not a command. That's the difference between parameterized queries and string concatenation.

Important Terms

  • INNER JOIN: Returns rows that have matches in both tables.
  • LEFT JOIN: Returns all rows from the left table, NULLs for missing right-side rows.
  • Alias: A short name assigned to a table inside a query (e.g., FROM users u).
  • Parameterized Query: A query with placeholders ($1, $2) filled in safely by the database driver at runtime.
  • SQL Injection: An attack where malicious SQL is embedded in user input and executed by the database.
  • ILIKE: Case-insensitive pattern matching using % wildcards.
  • OFFSET: Number of rows to skip before returning results — used for pagination.
  • RETURNING: Postgres clause that returns the affected rows immediately after INSERT or UPDATE.
  • to_jsonb(): Postgres function that converts a table row into a jsonb object.

Key Takeaways

  • Write queries starting with FROM — it forces clarity about where data comes from.
  • Use LEFT JOIN when the related record might not exist; INNER JOIN when it is guaranteed.
  • Always use parameterized queries for user-provided values — never string concatenation.
  • Use to_jsonb(row.*) to embed related table rows as JSON in a single query.
  • OFFSET = (page - 1) * limit. Convert from user's 1-based page numbers to 0-based offsets.
  • Use RETURNING * after INSERT/UPDATE to get the modified row without a second query.

Quick Revision

  • Build order: FROM → JOIN → WHERE → ORDER BY → SELECT.
  • INNER JOIN: both sides must match. LEFT JOIN: left side always returned.
  • SQL Injection: Use $1, $2 placeholders, never string concatenation.
  • ILIKE 'J%' → names starting with J, case-insensitive.
  • OFFSET = (page - 1) × limit.
  • INSERT ... RETURNING * → returns newly created row.
  • UPDATE ... SET bio=$1 WHERE id=$2 RETURNING * → partial update, returns updated row.
