Caching Strategies & Eviction Policies

When implementing a cache in your backend, you must decide how data gets into the cache (Strategies) and how data gets removed when the cache is full (Eviction Policies).

Caching Strategies

There are two primary patterns backend engineers use to read from and write to a cache.

1. Lazy Caching (Cache Aside)

In Lazy Caching, the cache isn't populated proactively. Data is only loaded into the cache when someone actually requests it.

Workflow:

  1. Client requests data.
  2. Server checks the cache.
  3. Cache Hit: Return data to client.
  4. Cache Miss: Server fetches data from the database.
  5. Server stores that data in the cache.
  6. Server returns data to client.

Advantage: You never waste memory caching data that users aren't asking for.

2. Write-Through Caching

In Write-Through, the focus is on keeping the cache perfectly synchronized with the database during write operations (POST, PUT, PATCH).

Workflow:

  1. Client submits an update (e.g., changing their profile bio).
  2. Server updates the database.
  3. In the exact same execution flow, the server updates the cache with the new bio.
  4. Server returns success to client.

Advantage: The cache is never stale.
Disadvantage: Write operations take slightly longer because you are writing to two places simultaneously (overhead).

Eviction Policies

Because RAM is limited, your cache will eventually fill up. When a new piece of data needs to be stored, the cache must decide which old piece of data to delete. This is called an Eviction Policy.

Policy How it Works When to Use
No Eviction If the cache is full, it throws an "Out of Memory" error on the next insert. Rarely used, mostly for strict data sets where dropping data is unacceptable.
LRU (Least Recently Used) Tracks the timestamp of when a key was last accessed. Drops the key that hasn't been touched in the longest time. Standard default for most applications. (e.g., dropping yesterday's trending topic for today's).
LFU (Least Frequently Used) Tracks the number of times a key has been accessed. Drops the key with the lowest access count, regardless of when it was last touched. When some data is inherently more popular overall than others and must be preserved.
TTL (Time to Live) Each key is given a countdown timer (e.g., 60 minutes). When the timer hits zero, the key is automatically invalidated and deleted. For data with a known expiration window (e.g., an auth token valid for 24 hours).

Mental Model

Lazy Caching: Going to the grocery store only when you check the fridge and realize you are out of milk.

Write-Through: Buying a carton of milk every single time you use a splash for your coffee, ensuring the fridge is always 100% full.

Eviction (LRU): Cleaning out your closet by throwing away the shirt you haven't worn in 5 years.

Important Terms

  • Cache Aside: Another term for Lazy Caching. The app acts as a middleman between cache and DB.
  • Stale Data: Data in the cache that no longer matches the primary database (e.g., the cache says a product costs $10, but the DB was updated to $15).
  • Eviction: The forced removal of data to free up memory.

Key Takeaways

  • Lazy caching is efficient for memory but incurs a penalty on the first request (Cache Miss).
  • Write-through caching ensures data accuracy but adds latency to writes.
  • LRU is the most common eviction policy because recent usage is the strongest predictor of future usage.
