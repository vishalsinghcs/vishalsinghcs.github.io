What is a Database and DBMS?

Databases are the persistence layer of every backend system. Understanding what they are, how they differ from in-memory storage, and why dedicated database management software exists is the foundation for everything else.

Why Do We Need Databases?

At its core, a database is a system that persists information across sessions. Persistence means storing data in a way that survives even after the program that created it has been stopped, restarted, or even replaced.

Without persistence, every time a user opened a to-do app they would lose all their data — every task created would be gone. Persistence is what makes software useful in the real world. The information must remain in the same expected state even after a significant amount of time has passed, or across different physical locations.

The Broad Definition of "Database"

In the widest sense, any form of structured storage can be called a database:

  • A smartphone contact list
  • Browser localStorage / sessionStorage / cookies
  • A plain text file where you jot down notes

What all these have in common: they provide some way to create, read, update, and delete (CRUD) data.

However, when we say "database" in the context of backend systems and servers, we mean specifically disk-based databases — systems that store data on HDD or SSD storage.

Disk-Based vs. RAM-Based Storage

Two main types of storage are relevant to every backend system:

Characteristic RAM (Primary Memory) Disk (Secondary Memory)
Speed Very fast Relatively slow
Cost Expensive per GB Cheap per GB
Typical capacity 8 GB – 128 GB 512 GB – 2 TB+
Persistence Lost on shutdown Survives shutdown
Use case Caching (Redis) Databases (Postgres, MongoDB)

The key insight: databases choose disk storage because they need capacity and durability, accepting a speed trade-off. Caching systems like Redis choose RAM because they need speed, accepting the trade-off of smaller capacity and volatility.

What is a DBMS?

Simply storing bytes on a disk is not enough. We need software that can efficiently organize, retrieve, modify, and protect that data. This software is called a Database Management System (DBMS).

The core responsibilities of a DBMS are:

  • Data Organization — Organizing data so that all CRUD operations (create, read, update, delete) are efficient, even at hundreds of GB of scale.
  • Access — Providing a standard interface (typically SQL) through which clients can perform all necessary operations.
  • Integrity — Ensuring the accuracy and validity of data. For example, a DBMS can enforce that a payment_amount field only ever stores numbers, not strings. If bad data is pushed, the DBMS rejects it at the database level before it corrupts your records.
  • Security — Protecting data from unauthorized access through users, roles, and permissions.

Why Not Just Use Text Files?

Before DBMS software existed, developers stored all data in text files. This approach failed for three fundamental reasons:

1. Parsing

To find a specific customer in a text file, you must write application code (in JavaScript, Python, Go, etc.) to read the file, split every line, and compare each field one by one. This is slow and extremely error-prone — a parsing bug can corrupt your entire dataset.

2. No Structure

A text file has no schema. You cannot enforce that a payment field must only contain numbers. Any arbitrary text can be placed anywhere. Consistency of data across thousands of entries is impossible to guarantee.

3. Concurrency

What happens when two users try to modify the same record simultaneously? With a text file, both read the same value (e.g., amount = 40). One increases it to 60, the other decreases it to 20. The result depends on which write operation hits the disk last — completely non-deterministic. A DBMS manages concurrent access with locking, transactions, and isolation mechanisms to guarantee consistent results.

Relational vs. Non-Relational Databases

At the highest level, DBMS software falls into two major categories:

Feature Relational (SQL) Non-Relational (NoSQL)
Data structure Tables, rows, columns Documents, collections
Schema Strict, predefined Flexible / schema-less
Data integrity Enforced at DB level Application must enforce it
Query language SQL (Structured Query Language) Varies (MongoDB Query Language, etc.)
Relationships Foreign keys, JOINs Embedded documents / manual
Examples PostgreSQL, MySQL, SQL Server MongoDB, CouchDB, DynamoDB

When to Choose Relational

Use a relational database when your data is structured and consistency matters. A CRM (Customer Relationship Management) system is a classic example — it must maintain accurate, consistent data about customers, contacts, and sales opportunities. Strong data integrity and complex query capabilities (JOINs across tables) are essential.

When to Choose Non-Relational

Use a non-relational database when your data is inherently unstructured or highly variable. A CMS (Content Management System) is a good example — blog articles can contain text, images, embedded videos, code blocks, or YouTube embeds. The schema is unknown upfront. MongoDB's flexible document model suits this case.

The Trade-off With Non-Relational

NoSQL databases offer flexibility, but they push the responsibility for data integrity to the application layer. Your backend code must validate consistency that a relational DBMS would enforce automatically. Application code changes frequently — which makes it far easier to introduce bugs that silently corrupt data over time.

Mental Model

A DBMS is a librarian, not just a building. The library (disk) stores books (data). But without a librarian (DBMS), you'd have to walk every single aisle to find a specific book. The librarian maintains a catalogue (index), enforces rules about what books belong where (schema/integrity), controls who can access restricted sections (security), and ensures two people don't accidentally tear the same page (concurrency).

Important Terms

  • Persistence: Storing data so it survives after the program that created it has stopped.
  • DBMS: Database Management System — software that manages storage, retrieval, and integrity of data.
  • Disk-based storage: Secondary memory (HDD/SSD) — large, cheap, slower.
  • RAM: Primary memory — fast, expensive, volatile (data lost on shutdown).
  • Data Integrity: The guarantee that data is accurate, valid, and consistent at all times.
  • Concurrency: Multiple users or processes accessing/modifying the same data simultaneously.

Key Takeaways

  • Databases persist data so it survives program restarts and time.
  • In backend systems, "database" means disk-based storage — not RAM.
  • Caching (Redis) uses RAM for speed. Traditional databases (Postgres, MongoDB) use disk for capacity and persistence.
  • Text files fail as databases due to poor parsing performance, no schema enforcement, and no concurrency handling.
  • Relational DBs enforce data integrity at the DB level. Non-relational DBs push that to application code — which is riskier.

Quick Revision

  • Database: Any persistent, structured storage with CRUD operations.
  • DBMS: Software managing organization, access, integrity, security.
  • RAM vs Disk: RAM = fast, small, volatile. Disk = slow, large, durable.
  • Text file failures: Slow parsing, no schema, no concurrency.
  • Relational: Tables, strict schema, SQL, strong integrity (Postgres, MySQL).
  • Non-relational: Documents, flexible schema, integrity in app code (MongoDB).
  • CRM → Relational. CMS → Non-relational.
