Global Error Handling

Global error handling is the practice of catching all errors in a single, centralized middleware instead of scattering try/catch blocks indiscriminately across your codebase. This acts as a final safety net for your backend architecture.

The Typical Backend Architecture

To understand where errors should be handled, we first define the layers of a typical request lifecycle:

  1. Routing Layer: Decides which handler processes the incoming request.
  2. Handler Layer: Extracts payload data, performs validation, and formats the final HTTP response. Throws Validation Errors (400 Bad Request).
  3. Service Layer: The orchestrator for business logic. It can call multiple repository methods. Throws Logic Errors.
  4. Repository Layer: The leaf node executing single database queries. Throws Database Errors (e.g., Unique Constraint Violations, No Rows Found).

Not all errors should be resolved the moment they occur. The key concept is propagation: catching lower-level exceptions, wrapping them with necessary context, and bubbling them up to higher levels.

If the Repository Layer encounters a database error, it shouldn't try to send an HTTP response. Instead, it bubbles the error up to the Service Layer, which bubbles it to the Handler Layer, until it reaches the Global Error Handler.

The Centralized Middleware

The Global Error Handler sits as a middleware at the very edge of your application. When an error bubbles up, it inspects the error type and formats an appropriate response:

  • Validation Error (Handler): Returns 400 Bad Request with a list of malformed fields.
  • No Rows Returned (Repository): Returns 404 Not Found.
  • Foreign Key Violation (Repository): Returns 404 Not Found (the referenced resource doesn't exist).
  • Unique Constraint Violation (Repository): Returns 400 Bad Request (e.g., "Email already exists").
  • Unknown Error: Returns 500 Internal Server Error with a generic message like "Something went wrong."

Advantages

This approach prevents code redundancy (no duplicate error checking in every controller). Most importantly, it is highly robust. By routing all errors through one gateway, you guarantee no unhandled exception will leak internal system details to a user.

Mental Model

Think of your architecture as a corporate ladder. A worker (Repository Layer) doesn't issue public press releases (HTTP Responses). They pass the problem up to the CEO (Global Middleware), who controls exactly what the public sees.

Important Terms

Propagation: The act of passing an error up the call stack to be handled by a higher-level function.

Middleware: A centralized function that sits between incoming requests and outgoing responses, intercepting them for tasks like error formatting.

Key Takeaways

  • Never send HTTP responses directly from deep business logic or database layers.
  • A centralized error handler eliminates duplicate code and ensures consistency.

Quick Revision

  • Route -> Handler -> Service -> Repo.
  • Bubble errors up the stack.
  • Middleware converts internal errors to HTTP status codes.
