Pagination, Sorting and Filtering

Every list API should support three capabilities: limiting how many results come back at once (pagination), controlling the order of results (sorting), and narrowing results to a specific subset (filtering). Without these, list APIs become a performance liability at scale.

Why Pagination Exists

Consider a database with 100,000 organizations. If a client calls GET /organizations and the server returns all 100,000 records in a single response:

  • JSON serialization of 100,000 records is a CPU-intensive operation on the server.
  • Network transfer of that payload causes significant latency.
  • The client (browser or mobile app) cannot display 100,000 records at once — users will only see the first 10-20 on screen anyway.

Pagination solves this by returning data in fixed-size chunks (pages). The client fetches the first page, and requests subsequent pages only when the user scrolls or clicks "next".

Pagination Parameters

Since list APIs use GET, data must be sent from client to server as query parameters (not a request body). The two standard pagination parameters are:

Parameter Meaning Default (if not sent)
page Which chunk of data to return (1-indexed) 1 (first page)
limit How many records per page 10 or 20 (server-defined)

Important: If the client does not send these parameters, the server must apply sensible defaults. Never return a validation error for missing pagination params — they are optional and have defaults.

Pagination Response Structure

A paginated list response should always include metadata so the client can determine how to navigate further:

{
  "data": [ ... ],          // The actual records for this page
  "total": 50,              // Total count of ALL records in the database (ignoring pagination)
  "page": 2,                // Which page this response is for
  "totalPages": 5           // Total number of pages (ceil(total / limit))
}

The total and totalPages fields allow the frontend to:

  • Display "Showing 10 of 50 results" UI
  • Render page number buttons (1, 2, 3, 4, 5)
  • Stop making API calls for infinite scroll when page === totalPages

Sorting

Without explicit sorting, a database returns records in an indeterminate order — this order can change between calls, causing flickering and inconsistent UIs. The server should always sort data before returning it, and should support client-controlled sorting via query parameters.

Default Sort Behaviour

The server should always sort by createdAt in descending order by default. This is the most natural order for most list views — the user expects to see the newest entries first.

Sorting Parameters

Parameter Meaning Default
sortBy Which field of the resource to sort by createdAt
sortOrder Ascending or descending desc (descending)

Example: GET /organizations?sortBy=name&sortOrder=asc returns all organizations sorted alphabetically A→Z.

Even if the client only sends sortBy=name without sortOrder, the server should default sortOrder to desc.

Filtering

Filtering allows clients to narrow the result set to entries that match a specific condition. Like pagination and sorting, filters are passed as query parameters.

Example Request What it Returns
GET /organizations?status=active Only organizations with status = "active"
GET /organizations?status=archived Only organizations with status = "archived"
GET /organizations?name=Acme Only organizations where name matches "Acme"

Filters, pagination, and sorting can all be combined in a single request: GET /organizations?status=active&sortBy=name&sortOrder=asc&page=2&limit=10

The "Sensible Defaults" Principle

This is one of the most important API design rules in the lecture: the server should not demand that clients send obvious information. If it is obvious what the default behaviour should be, the server should implement it as a default and make the parameter optional. This reduces client-side complexity and makes the API feel intuitive to use.

Mental Model

Pagination is like a library catalogue. You don't carry all 100,000 books home at once — you flip through a catalogue (paginated list) and request only the books you need. The catalogue shows you "Page 3 of 50, 1,000 total books" — that's your page, totalPages, and total metadata.

Important Terms

  • Pagination: Splitting large list results into smaller, fixed-size chunks fetched one at a time.
  • page: Which chunk to return (1-indexed).
  • limit: How many records per chunk.
  • total: Total count of all matching records in the database.
  • totalPages: Math.ceil(total / limit)
  • Default Sort: createdAt DESC — most recent first.

Key Takeaways

  • All list APIs should support pagination, sorting, and filtering via query parameters.
  • If client doesn't send page or limit, server provides sensible defaults (page=1, limit=10).
  • Default sort is always createdAt DESC. Without explicit sorting, DB results are non-deterministic.
  • A paginated response always includes: data, total, page, totalPages.

Quick Revision

  • page: Which page, default 1.
  • limit: Records per page, default 10/20.
  • sortBy: Field to sort by, default createdAt.
  • sortOrder: asc or desc, default desc.
  • Filter params: Pass field name + value as query param.
  • Empty result: Return { data: [], total: 0 } with 200.
