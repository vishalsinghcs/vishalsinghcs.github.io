Interview Questions

10 - Caching: Network Caching, In-Memory DBs, Caching Strategies, Eviction Policies, and Backend Use Cases.

These questions test your ability to reason about caching architectures, memory management, data consistency, and edge-case failures in a high-traffic production system. The goal is engineering thinking, not recitation of terms.

1. Scenario-based Questions

  • Your Redis cache goes down in production entirely. Because you implemented a Cache Aside (Lazy) strategy, the application gracefully falls back to querying the primary PostgreSQL database. However, within 60 seconds, your entire platform crashes. Walk through the cascade of failures that caused the crash, and how you would redesign the system to survive a complete cache failure.
  • You deployed a critical bug fix to your frontend CSS file. Users are complaining that the UI looks completely broken. You check the server, and the new file is definitely deployed. You realize the CDN edge servers are serving the old CSS file. What exactly is happening, why did the CDN do this, and what is the immediate fix and long-term architectural solution?
  • You implement rate limiting using Redis. A middleware checks the X-Forwarded-For header and increments a counter. Suddenly, your customer support gets flooded with complaints that every single user from a specific corporate office is being blocked with a 429 Too Many Requests error, even though no individual user is exceeding the limit. What is the networking root cause of this, and how do you fix your rate limiter?
  • A celebrity makes a viral post on your platform, instantly driving 500,000 users to view it simultaneously. The post is not in the cache yet. Due to your Lazy Caching strategy, 500,000 requests miss the cache and hit the database to compute the post's stats at the exact same millisecond. This completely locks the database (a Cache Stampede). How would you modify your caching strategy to prevent this specific scenario?
  • Your startup decides to switch from Lazy Caching to Write-Through caching for user profiles to ensure data is never stale. A week later, latency metrics for the "Update Profile" endpoint show an increase from 50ms to 400ms. What are the specific architectural reasons why a Write-Through cache slows down write operations, and under what conditions is this trade-off acceptable?
  • You configured Redis with a "No Eviction" policy. Over a period of six months, the 16GB of RAM gradually fills up to 100%. What exactly happens to your backend application the moment a user attempts to log in and create a new session token, and how does this contrast with what would happen under an LRU policy?
  • Your infrastructure team migrates your primary API servers to a new AWS region, completely changing their IP addresses. The DNS records are updated successfully. However, 15% of your global users report they cannot access the app for the next 24 hours. Walk through the hierarchy of DNS caching (OS, Browser, Resolver) to explain why this happened and what DNS configuration you missed prior to the migration.
  • An external Weather API charges you $0.01 per request and allows 1000 requests per minute. You implement API caching with a 60-minute TTL. Your bill for the month comes back at $50,000 (meaning millions of requests were made). Upon investigation, you realize you cached the data based on the exact latitude and longitude down to 8 decimal places. Why did this caching strategy fail to protect your billing, and how would you redesign the cache key structure?

2. "What if?" Questions

  • What if hardware RAM was completely non-volatile (persistent) and as cheap as Disk storage, allowing you to store petabytes of data in memory? How would this fundamentally change the way we architect databases and caching layers? Would we still need Redis?
  • What if CDNs had no concept of TTL (Time to Live) and simply cached everything forever until manually purged? Trace the operational nightmares this would cause for a heavily dynamic platform like Amazon or Twitter, specifically focusing on deployments, data accuracy, and edge node storage capacity.
  • What if DNS caching didn't exist at the Operating System or Browser level, and every single HTTP request required a full recursive resolution starting from the Root servers? Explain the impact on global internet latency, the load on top-level domain servers, and the potential security vulnerabilities.
  • What if the only caching strategy available was Write-Through, and Lazy Caching (Cache Aside) was impossible? Explain which specific types of applications would become financially and computationally unscalable, and why.

3. Debugging Questions

  • Users are reporting a severe security issue: after logging in, they occasionally see the profile dashboard of completely random users. You are using Redis to store session tokens, mapping `session_id -> user_id`. You verify the token generation is perfectly secure. What is the most likely bug in how your middleware reads or writes to the cache that would cause cross-contamination of sessions under high concurrency?
  • You implement a Redis-based rate limiter using a simple key-value counter (IP_ADDRESS: count) with a 60-second TTL. However, malicious bots are successfully making thousands of requests per minute without getting blocked. When you check Redis, the counter for their IP never goes above 1, but the TTL keeps resetting to 60. What is the logic flaw in how your code interacts with Redis?
  • Your monitoring tools show that the primary Postgres database is completely idle, yet users are experiencing 500 Internal Server Errors when trying to load their dashboards. The cache server is maxed out on memory and CPU. Given that you use a Lazy Caching strategy, trace the request path to identify where the failure is occurring and why the fallback to the database isn't happening.

4. Design Questions

  • Design a "Trending Topics" caching system for a platform like X (Twitter). The raw data is millions of tweets per minute. You need to serve the top 10 trends to users instantly. Detail the architecture: How is the computation offloaded? How does data get into the cache? What is the cache key structure? What eviction policy is used, and how do you ensure the UI updates seamlessly without users seeing stale trends?
  • Design a CDN caching strategy for a global e-commerce site. You have static assets (images, JS, CSS), semi-dynamic data (product catalogs, pricing during a sale), and highly dynamic data (user shopping carts, checkout sessions). Detail exactly how you would configure edge caching rules, TTLs, and cache-busting strategies for each of these three data types to balance speed with perfect data accuracy.
  • Design an external API rate limit cache. Your backend consists of 50 horizontally scaled Node.js servers, all making requests to a 3rd party AI API that absolutely hard-limits you to exactly 500 calls per minute globally. If you hit 501, your account is banned. Detail how you would use Redis to design a distributed, atomic counter that guarantees no race conditions across the 50 instances can breach the 500 limit.

5. Architecture & Trade-off Questions

  • A junior engineer asks: "Instead of running a separate Redis server which adds network latency, why don't we just store the cached data in a Javascript Map() or object directly in the Node.js memory?" Compare and contrast in-memory application variables vs a dedicated Redis cache regarding horizontal scaling, persistence, memory limits, and state sharing. When is the junior engineer's approach actually the right choice?
  • You are architecting the caching layer for two different features: (A) The checkout inventory system where product stock counts must be perfectly accurate, and (B) The comments section on a blog post. Analyze the trade-offs between Lazy Caching and Write-Through caching for both scenarios. Defend your choice of strategy for each feature based on consistency requirements and write-heavy vs read-heavy patterns.
