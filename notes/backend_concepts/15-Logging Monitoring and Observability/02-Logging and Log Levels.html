Logging and Log Levels

Recording crucial events throughout an application's lifecycle.

Logging is the practice of maintaining a journal or diary of all important, suspicious, and security-related events in your backend application. When the time comes to debug an issue, logs tell us exactly what happened, when it happened, and the context surrounding it.

Context is provided through metadata attached to a log line—such as user ID, the latency of a request, the specific HTTP method, and the unique request ID.

Standard Log Levels

We categorize logs by severity. A well-configured application will only output certain levels depending on the environment (e.g., hiding debug logs in production).

  • Debug: Overwhelmingly detailed logs used strictly during development to troubleshoot logic. Disabled in production to reduce noise.
  • Info: General operational logs indicating a successful flow. For example, "User account created."
  • Warn: Suspicious or faulty behavior that isn't the application's fault, such as a user typing an incorrect password or an incoming invalid payload.
  • Error: Clear failures, such as a database query timing out or an external API going down.
  • Fatal: Critical bugs causing the application process to forcibly crash and restart.

Structured vs. Unstructured Logs

The format of your logs is just as important as the content. We use different formats for humans versus machines.

Unstructured Logs (Console)

During local development, we want logs to be printed in plain text with colors. This makes them highly human-readable, allowing developers to spot errors at a glance inside their VS Code terminal.

Structured Logs (JSON)

In a production environment, reading logs line-by-line is impossible. Logs are ingested by log management stacks (like ELK or Grafana Loki). If you use plain text in production, these tools struggle to parse and extract valuable parameters.

Therefore, production logs are printed in JSON format. JSON allows log aggregators to easily index fields like `userId` or `statusCode`, making your logs highly searchable.

Mental Model

Logging levels are like filters. In development, you want the net wide open (Debug) to catch everything. In production, you tighten the net (Info and above) so you don't drown in unnecessary data.

Important Terms

Metadata: Additional contextual data appended to a log (e.g., timestamps, User IDs, latencies).

Structured Logging: Outputting logs in machine-readable formats (like JSON) so external aggregators can parse and index the variables.

Key Takeaways

  • Always attach metadata to logs for better debugging context.
  • Use Debug strictly for development; keep production noise low.
  • Never use plain text logs in production; they break parsers.

Quick Revision

  • Debug: High detail, dev only.
  • Info: Standard operations.
  • Warn: User errors, unusual state.
  • Error: System failures.
  • Fatal: App crashes.
