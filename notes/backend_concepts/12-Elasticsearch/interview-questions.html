Interview Questions

12 - Elasticsearch: Full Text Search, Inverted Indexes, Relevance Scoring, and Performance.

These questions evaluate your deep understanding of how search engines operate at scale, indexing trade-offs, and architectural decision-making. The goal is engineering thinking, not recitation of terms.

1. Scenario-based Questions

  • You have a Postgres database with 50 million product records. The PM wants a fast autocomplete search bar with typo tolerance. How do you implement this, and what architectural changes are required?
  • Users complain that when they search for "running shoes", they see socks with the description "best for running" before the actual shoes. How would you solve this relevance issue using field boosting?
  • A user searches for "IPHONE" but all documents are stored as "iphone". How does Elasticsearch handle case sensitivity to ensure a match is still found?
  • Your search results for a common term like "machine" are burying documents that have "Machine" as the exact title, favoring documents that mention "machine" 10 times in a long description. How does the BM25 algorithm address this automatically via document length?
  • You are using the ELK stack to manage logs for 20 microservices. Developers complain they can't see the latest logs for the last 15 minutes, but the application is running fine. What components of the ELK stack would you investigate first and why?
  • A client requests that when a user searches for "laptpo", it should still return "laptop". Explain how you would implement this using fuzzy search and the concept of edit distance.
  • You perform a bulk insertion of 10,000 documents into Elasticsearch. A query executed 10 milliseconds later returns 0 results, but a query executed 2 seconds later returns all 10,000. Why does this happen, and how does Elasticsearch handle index refreshing?
  • Your company is using Elasticsearch as the primary datastore instead of Postgres. You receive a requirement to join a `users` table with an `orders` table to perform a complex search. How do you approach this in a document-based search engine?

2. "What if?" Questions

  • What if you define a field as a `keyword` type instead of a `text` type? How does this drastically alter how the inverted index stores the data, and what queries will now fail?
  • What if a user searches for a stop-word like "the" or "is" across millions of documents? How does the Document Frequency (DF) parameter in BM25 prevent the search score from becoming meaningless?
  • What if Elasticsearch used standard B-Tree indexing (like a relational database) instead of an inverted index? How would this impact the performance of full-text search?
  • What if you configure an Elasticsearch cluster to synchronously accept writes directly from the main API without a message broker? What happens during a temporary network partition?

3. Debugging Questions

  • You notice that Elasticsearch search latency has suddenly spiked from 50ms to 5 seconds. CPU usage is maxed out, but the search volume hasn't changed. What indexing or ingestion behaviors might be causing this performance degradation?
  • A developer accidentally indexed a 100MB PDF document into a single text field of a single document. Explain how this heavily penalizes the document's BM25 score for any standard query.
  • You run a search for "Apple iPhone" and expect the results to only include documents containing both words. Instead, you get millions of results, some only containing "Apple" (the fruit) and some only containing "iPhone". What went wrong with your query DSL structure?

4. Design Questions

  • Design a data synchronization pipeline that keeps an Elasticsearch index eventually consistent with a primary Postgres database without significantly impacting the latency of the main API's write operations.
  • Design an autocomplete (type-ahead) feature for a global e-commerce site. Explain how you would model the data differently than a standard full-text search (e.g., using edge n-grams).
  • Your application generates 1TB of logs per day. Design an ELK stack architecture to ingest, process, and query these logs efficiently, taking into account data retention and index lifecycle management.

5. Architecture & Trade-off Questions

  • A team proposes using Postgres full-text search instead of deploying a new Elasticsearch cluster for a read-heavy blog platform. Under what specific conditions is this architectural choice superior, and when does it become a liability?
  • Why is the write (ingestion) phase in Elasticsearch so highly CPU-intensive compared to appending a row in a standard relational database? What trade-off is being made here regarding read vs. write performance?
