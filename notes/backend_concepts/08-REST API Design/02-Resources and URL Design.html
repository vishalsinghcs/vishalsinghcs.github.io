Resources and URL Design

Resources are the nouns of REST — everything your API exposes is a resource. URL design is how you give those resources a clean, hierarchical, and consistent address.

Identifying Resources

The starting point for any API design is identifying your resources. The practical rule: look at your wireframes and requirements, and pull out all the nouns. For a project management platform similar to Jira or Linear, the nouns are immediately visible:

  • Organizations — top-level entity; users belong to organizations
  • Projects — scoped under an organization
  • Tasks — scoped under a project
  • Users — accounts that can belong to organizations
  • Tags — applied to tasks for categorization

Once you have identified your resources, each one will have its own set of API endpoints. The patterns for those endpoints are repeatable — which is exactly what makes REST APIs learnable and predictable.

Rules for Path Segments

Rule 1: Always Use Plural Nouns

Even when fetching a single resource, the path segment representing the resource type must be plural. This is non-negotiable in REST standards. The resource type refers to the collection, not the single instance.

WrongCorrect
/organization/organizations
/book/123/books/123
/user/profile/users/profile

Rule 2: Always Lowercase

URLs travel across different environments — different operating systems, servers, and clients may handle case differently. Always use lowercase in path segments to eliminate case-mismatch bugs entirely.

Rule 3: Use Hyphens, Not Spaces or Underscores

If a path segment is a multi-word phrase, replace spaces with hyphens (-). Never use spaces (they are URL-encoded to %20) or underscores (some environments treat them as invisible in links).

InputSlug
Harry Potterharry-potter
Active Tasksactive-tasks

Rule 4: Forward Slash = Hierarchy

Every / in a URL path represents a hierarchical parent-child relationship. Reading a URL left to right should tell a clear story:

  • /organizations → The collection of all organizations
  • /organizations/abc123 → A specific organization
  • /organizations/abc123/archive → An action performed on that specific organization

This hierarchical clarity is what makes RESTful URLs intuitive — a reader can understand the relationship structure without documentation.

Rule 5: Use camelCase in JSON Fields

When it comes to the JSON payloads you send and receive, always use camelCase for field names. This is the universal JSON standard:

WrongCorrect
organization_idorganizationId
created-atcreatedAt
DSC (abbreviation)description

Resource Identification and Design Process

For each resource you've identified, the following workflow produces the full API surface:

  1. Name the resource as a plural noun (e.g., organizations)
  2. List the CRUD actions needed: create, list, get-one, update, delete
  3. Identify any custom actions that cannot be expressed as CRUD (e.g., archive, clone, send)
  4. Assign HTTP methods to each action (covered in the next note)
  5. Decide the request body fields, excluding server-generated fields (ID, createdAt, updatedAt)

Mental Model

Think of resources as physical file folders in a filing cabinet. The cabinet label is /organizations (plural — always). A specific folder inside is /organizations/abc123. A sticky note action attached to that folder is /organizations/abc123/archive. You read the path like reading a filing system — and the hierarchy is self-evident.

Important Terms

  • Resource: A noun — any identifiable data entity your API manages (user, order, project).
  • Path Segment: The portion of a URL between two forward slashes.
  • Dynamic Parameter: A variable in the URL that represents a specific resource ID (e.g., :id).
  • Slug: A human-readable, URL-safe version of a resource name (lowercased, hyphenated).
  • API Versioning: Adding /v1 to the URL path to allow breaking changes to be deployed without affecting existing integrations.

Key Takeaways

  • Resources = nouns extracted from your wireframes and requirements.
  • Path segments are always plural, lowercase, and hyphenated for multi-word phrases.
  • Forward slashes represent a hierarchical relationship between resources.
  • JSON fields always use camelCase. Never use abbreviations.
  • IDs, createdAt, updatedAt are server-generated and should never be accepted from client payloads.

Quick Revision

  • Resource extraction: Nouns from wireframes.
  • Always plural: /books not /book.
  • Always lowercase: /organizations not /Organizations.
  • Hyphens: /active-tasks not /active_tasks.
  • Hierarchy: / = parent-child relationship.
  • JSON keys: camelCase, no abbreviations.
