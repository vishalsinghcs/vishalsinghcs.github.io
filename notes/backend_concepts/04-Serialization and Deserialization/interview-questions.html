Interview Questions

04 - Serialization & Deserialization: Translating Data Across the Network

These questions test engineering thinking, debugging ability, production experience, and trade-off analysis regarding data parsing and formatting. Try to answer them thoroughly without looking up definitions.

1. Scenario-based Questions (40%)

  • You deploy a new microservice written in a strictly typed language (like Rust) that communicates with a legacy Node.js frontend via JSON. The frontend sends {"user_id": "123"} but the backend server panics and crashes when processing the request. How would you investigate and resolve this?
  • A high-throughput trading application is currently using JSON over HTTP for inter-service communication and is experiencing severe CPU bottlenecks primarily from parsing JSON strings. How would you approach solving this performance issue?
  • A mobile app team complains that the JSON responses from your API are too large and slow over 3G networks. Besides network-level compression (like Gzip), what serialization strategies could you use to shrink the payload size?
  • A frontend developer accidentally leaves a trailing comma in a JSON request body (e.g. [1, 2,]). Your API returns a 500 Internal Server Error, and the logs show a parsing failure deep in a framework library. How do you handle this gracefully to return a helpful 400 Bad Request instead?
  • Two internal services share a JSON schema. Service A adds a new field is_active to a model, but Service B suddenly crashes when it tries to deserialize the updated model, despite not needing the new field. What could be the root cause in Service B's deserialization logic?
  • You notice that a specific API endpoint that accepts heavily nested JSON payloads is vulnerable to memory exhaustion and CPU spiking attacks. How do you protect your deserialization layer?
  • Your team wants to migrate from JSON to Protobuf for all internal microservices to improve performance, but external public clients still need to consume JSON. How do you architect this transition without maintaining two completely separate codebases?
  • An API consumer reports that timestamps sent by your server (e.g., "2024-05-12T10:00:00Z") are deserializing incorrectly on their client side depending on their local timezone. What is the likely problem with how the JSON is being parsed or generated?

2. "What if?" Questions (20%)

  • What if JSON didn't rigidly enforce double quotes around keys and allowed single quotes or unquoted keys (like raw JavaScript)? How would that impact building cross-language parsers?
  • What if binary serialization formats (like Protobuf) were used for all public-facing REST APIs instead of JSON? What problems would third-party developers face?
  • What if serialization didn't exist at all, and we just sent raw memory buffers (like a C struct in RAM) directly over the network to other servers? Why wouldn't this work reliably?
  • What if JSON specifications allowed arbitrary functions or executable code to be serialized and passed across the wire? What are the architectural and security implications?

3. Debugging Questions (15%)

  • A client sends a payload containing a monetary value "price": 10.50, but your backend database stores it as 10.49999999999. Where could the serialization pipeline be failing, and how do you fix it?
  • Your web server receives a JSON payload, but when you try to access the data in your handler, the body is completely empty or undefined despite the network tab confirming the data was sent. What configuration is missing?
  • A backend serializes an object and sends it to another service. The receiving service throws a TypeError: Object of type null is not JSON serializable. If the payload looked fine over the network, where could the issue lie?

4. Design Questions (15%)

  • Design an API schema for a complex analytics dashboard that needs to serialize and send millions of nested data points to the client efficiently without crashing the browser's parser.
  • Design a serialization strategy for a real-time multiplayer game server where the network payload size must be kept under 50 bytes per tick to reduce latency.
  • Design a system that receives legacy XML payloads from a bank partner and forwards them as modern JSON to an internal service, ensuring that strict schema validation is not lost in translation.

5. Architecture & Trade-off Questions (10%)

  • Why might a company choose to use YAML extensively for configuration files (like CI/CD pipelines or Docker), but absolutely forbid its use for real-time network request payloads?
  • When deciding between JSON and Protobuf for a new microservice architecture, what specific trade-offs are you making regarding schema evolution and backwards compatibility?
