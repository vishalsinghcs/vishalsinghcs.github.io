Types of Validation

Understanding the four primary categories of validation constraints you can apply to incoming client data.

While validation rules can be as loose or as strict as your application demands, they generally fall into one of four primary categories.

1. Type Validation

The most fundamental validation. It simply checks if the provided data matches the expected primitive data type (String, Number, Boolean, Array, or Object/JSON).

  • Example: An API expects an age field to be a Number. If the client sends the string "twenty" or a Boolean true, the validation fails.
  • Array Example: If an API expects an array of tags, it checks both that the payload is an array and that every element inside that array is of the expected type (e.g., strings).

2. Syntactic Validation

Checks whether the provided data follows a specific structure or format. It assumes the Type Validation has already passed (e.g., it is indeed a string).

  • Emails: Checking if a string follows the standard email structure (contains an @ symbol, a domain, and a top-level domain like .com).
  • Phone Numbers: Checking if a string contains a valid country code followed by a 10-digit number.
  • Dates: Checking if a string is formatted in the expected ISO pattern (e.g., YYYY-MM-DD).

3. Semantic Validation

Checks whether the provided data actually makes logical sense in the context of the real world or the specific business logic.

  • Dates in Time: A user provides a perfectly formatted date string (passes syntactic validation), but the date is in the future. If the field is DateOfBirth, a future date makes no semantic sense and should be rejected.
  • Logical Boundaries: A user provides an age of 430. While it passes Type Validation (it is a number), a human cannot be 430 years old, so it fails semantic validation.

4. Complex (Cross-Field) Validation

Validations where the rules of one field depend entirely on the value of another field in the same payload.

  • Matching Fields: Ensuring that the passwordConfirmation field is strictly equal to the password field.
  • Conditional Requirements: If a user sets a field isMarried to true, then a secondary field partnerName becomes strictly required. If isMarried is false, partnerName is optional or ignored.

Validation Types Comparison

Type What it Checks Example Failure
Type Primitive data type matches (String, Number) Sending "true" instead of boolean true
Syntactic Follows structural format (Regex patterns) Sending user#domain.com instead of @
Semantic Logical/Real-world sense Date of birth in the year 2050
Complex Cross-field dependencies Passwords do not match

Mental Model

Think of it like filtering water: Type validation removes the large rocks (rejects wrong objects). Syntactic validation removes the sand (checks shape/format). Semantic validation removes the microscopic bacteria (checks if it actually makes sense to drink).

Important Terms

  • Syntactic: Pertaining to syntax, rules, and formatting structure.
  • Semantic: Pertaining to meaning, logic, and context.

Key Takeaways

  • Validations execute in a hierarchy. You cannot run a semantic check (like age < 120) until you've run a type check to ensure the input is actually a number.
  • Error messages act as passive documentation for developers trying to integrate with your API.

Quick Revision

  • Type: Is it a string/number/boolean?
  • Syntactic: Is the string formatted like an email?
  • Semantic: Does this email domain actually exist? Is this birth date in the past?
  • Complex: Do these two password fields match?
