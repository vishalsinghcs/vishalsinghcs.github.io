HTTP Headers

HTTP headers are key-value pairs that send metadata about a request or response. They provide a separate layer of information independent of the URL and the message body.

Why Do We Need Headers?

Instead of putting all data in the URL or the body, headers provide a quick, accessible way to check metadata without parsing the entire payload.

This separation of concerns makes HTTP adaptable and efficient, enabling servers and intermediaries (like proxies or firewalls) to route and process requests effectively.

Categories of Headers

1. Request Headers

Sent by the client to provide information about the request context and the client's capabilities.

  • User-Agent: Identifies the client (e.g., a specific browser, mobile app, or Postman).
  • Authorization: Sends credentials (like a Bearer token) to authenticate the user.
  • Accept: Specifies what kind of content the client expects (e.g., JSON, HTML, plain text).

2. General Headers

Used in both requests and responses to provide metadata about the message itself.

  • Date: The timestamp of the message.
  • Cache-Control: Dictates caching mechanisms (e.g., no-cache or max-age).
  • Connection: Connection management (e.g., keep-alive or close).

3. Representation (Entity) Headers

Describe the specific payload/body being transmitted.

  • Content-Type: The media type of the body (e.g., application/json).
  • Content-Length: The size of the resource in bytes.
  • Content-Encoding: Specifies any compression applied (e.g., gzip or deflate).
  • ETag: A unique identifier used mostly for caching validation.

4. Security Headers

Used to enhance security by controlling browser behaviors.

  • Strict-Transport-Security (HSTS): Forces the client to communicate only over HTTPS.
  • Content-Security-Policy (CSP): Restricts where content (like scripts/images) can be loaded from, mitigating XSS.
  • X-Frame-Options: Prevents the page from being embedded in iframes (mitigates clickjacking).
  • X-Content-Type-Options: Prevents the browser from guessing the MIME type (MIME sniffing).

Capabilities Unlocked by Headers

Extensibility

HTTP is highly extensible because headers can be easily added or customized without altering the underlying protocol structure. Developers can even create custom headers (e.g., X-Custom-Header) for app-specific use cases.

Remote Control

Headers act as a remote control, allowing the client to send instructions influencing how the server responds:

  • Content Negotiation: Using Accept, Accept-Language, or Accept-Encoding.
  • Caching Control: Telling the server or intermediaries how to handle caching.

Mental Model: The Parcel

Think of an HTTP message as a parcel. You write the recipient address, phone number, and weight on the outside (Headers) so couriers can process it quickly. You don't put that metadata inside the box (Body), because forcing everyone to open the box to figure out where it goes is highly inefficient.

Quick Revision

  • Headers are metadata key-value pairs.
  • Request: Info about client (User-Agent, Accept).
  • Representation: Info about the body (Content-Type, Content-Length).
  • Security: Browser protections (CSP, HSTS).
  • They allow HTTP to be highly extensible and controllable.
