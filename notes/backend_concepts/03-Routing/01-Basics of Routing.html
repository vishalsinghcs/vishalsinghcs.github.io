Basics of Routing

If HTTP methods describe the "what" of a request, routing describes the "where". Routing maps client intentions to specific server-side logic.

Understanding the Concept

In backend architecture, a client request consists of two primary semantic components that determine how the server reacts:

  • The Method (The What): The HTTP method (GET, POST, PUT, DELETE) that expresses your intent or action (e.g., fetching data vs. creating data).
  • The Route (The Where): The URL path (e.g., /api/users) specifying the target resource.

Routing is simply the process of taking this combination—the method and the path—and mapping it to a specific set of instructions (a Handler) on the server. This handler then executes business logic, interacts with the database, and returns a response.

Static Routes

The simplest form of routing is the Static Route. As the name implies, these routes have no variable parameters; they are constant strings that never change.

Characteristics of Static Routes

  • They consist of fixed URL paths, such as /api/books or /dashboard/stats.
  • They don't contain dynamic placeholders.
  • They generally return or operate on a uniform structure or a bulk collection of data.

Example Workflow

Consider the static route /api/books:

  1. A GET request to /api/books might map to a handler that fetches an array of all books from the database.
  2. A POST request to the same /api/books route will map to an entirely different handler responsible for creating a new book resource.

Even though the string /api/books is identical in both cases, the server combines the method and the route as a unique "key". These two combinations will never clash, allowing the server to confidently route the requests to two completely separate logic blocks.

flowchart TD Route[Route: /api/books] Route -->|Method: GET| H1(getBooksHandler) Route -->|Method: POST| H2(createBookHandler) H1 -->|Returns| R1[Array of Books] H2 -->|Returns| R2[New Book Object]

Mental Model

Think of the server as a large office building. The Route is the room number, and the Method is the action you want to perform once you get there. Walking into Room 101 to pick up a document (GET) is completely different from walking into Room 101 to drop off a new document (POST).

Important Terms

  • Handler: The specific server-side function or block of code that executes when a route and method are successfully matched.
  • Resource: The object or data entity represented by the route (e.g., "books" in /api/books).

Quick Revision

  • Method = The Action/Intent (What).
  • Route = The Address (Where).
  • Routing maps Method + Route -> Handler.
  • Static routes are fixed strings with no variables.
  • Same routes with different methods map to entirely different handlers.
