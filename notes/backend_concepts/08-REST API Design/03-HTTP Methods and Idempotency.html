HTTP Methods and Idempotency

HTTP methods are the verbs of REST. Each method carries a semantic meaning that tells the server — and any API consumer — what kind of operation is being performed. Idempotency defines whether calling the same operation multiple times is safe.

The Five Core HTTP Methods

While HTTP defines many methods (HEAD, OPTIONS, CONNECT, TRACE…), five are used in everyday REST API design:

Method Operation Idempotent? Request Body? Typical Status Codes
GET Fetch / Read ✅ Yes No 200, 404
POST Create / Custom Action ❌ No Yes 201, 200
PUT Full Replace ✅ Yes Yes (full resource) 200
PATCH Partial Update ✅ Yes Yes (partial fields) 200
DELETE Delete ✅ Yes No 204

Idempotency Explained

Idempotency means: performing the same operation multiple times produces the same effect as performing it once. The key word is "side effect" — what changes in the server environment as a result of the call.

GET — Idempotent

Calling GET /books one time or a thousand times makes no change to the server. You are only reading data. The server state is unaffected by your calls. Idempotent by definition.

PATCH and PUT — Idempotent

You send PATCH /users/1 with {"name": "Alice"}. On the first call, the name changes from "Bob" to "Alice". On the second, third, and millionth call with the same payload, the name is still "Alice" — no additional change occurs. The side effect (name = Alice) is identical every time. Idempotent.

DELETE — Idempotent

You send DELETE /users/1. First call: user is deleted. Second call: the server returns a 404 because the user no longer exists. But crucially — no new side effect occurred. The state of the system (user doesn't exist) is the same whether you called it once or a thousand times. Idempotent.

POST — Non-Idempotent

You send POST /books with {"name": "Dune"}. First call: creates book with ID 1. Second call with same payload: creates another book with ID 2. Third call: ID 3. Each call produces a new and different side effect. This is why POST is the only non-idempotent method.

PUT vs. PATCH — When to Use Which

This is one of the most commonly confused distinctions in REST:

Method Payload Required Behaviour Use When
PUT All fields of the resource Completely replaces the resource with the provided payload Replacing a full resource (rare in modern SPAs)
PATCH Only the fields you want to change Merges changes into the existing resource Updating specific fields (preferred in modern JSON-heavy APIs)

In practice, modern single-page applications almost always use PATCH for updates. You rarely want to send an entire resource back just to change one field. The lecturer's guidance: "Stick to PATCH for partial updates. PUT is more relevant to the multi-page application era."

POST for Custom Actions

REST defines GET, PUT, PATCH, DELETE for structured data operations. But what happens when you need to perform an action that doesn't map cleanly to any of these — like archiving an organization, cloning a project, or sending an email?

The REST specification makes POST open-ended for exactly this reason. When an operation:

  • Is not a fetch (not GET)
  • Is not a standard create
  • Is not an update or delete
  • Is a verb action you want the server to perform

...use POST with the action name appended to the URL (e.g., POST /organizations/abc123/archive). The response code will depend on what the server does — it won't always be 201.

The "Other" Methods

  • HEAD: Like GET but only retrieves headers, not the body. Used to check if a resource exists or get its metadata cheaply.
  • OPTIONS: Used by browsers during the CORS preflight process. The browser asks: "Is my origin allowed to make this request?" The CORS middleware responds with the allowed origins and methods.

Mental Model

Think of idempotency like a light switch. Flipping it ON once or ten times — the light is still ON. That's idempotent. Now think of POST like pressing a doorbell. Press it once and one chime plays. Press it ten times and ten chimes play. Each press produces a new event. Not idempotent.

Important Terms

  • Idempotency: Performing the same operation N times produces the same side effects as performing it once.
  • Side Effect: Any change made to server state as a result of an API call.
  • PATCH: Partial update — only fields you send are changed.
  • PUT: Full replace — the entire resource is replaced with the payload.
  • Custom Action: A verb operation (archive, clone, send) expressed as POST /resource/:id/action.

Key Takeaways

  • GET, PATCH, PUT, DELETE are all idempotent. POST is the only non-idempotent method.
  • Use PATCH (not PUT) for partial updates in modern JSON-based APIs.
  • When an operation doesn't fit CRUD, use POST with a named action in the URL.
  • POST response codes are not always 201 — custom actions can return 200 if no new resource is created.

Quick Revision

  • GET: Read, idempotent, no body.
  • POST: Create or custom action, NOT idempotent, has body.
  • PUT: Full replace, idempotent, full body required.
  • PATCH: Partial update, idempotent, partial body.
  • DELETE: Delete, idempotent, no body.
  • HEAD: Metadata only.
  • OPTIONS: CORS preflight.
