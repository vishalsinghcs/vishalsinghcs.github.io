PostgreSQL and Data Types

PostgreSQL (Postgres) is the industry's preferred open-source relational database — and for good reason. Understanding why it wins and which data types to use for which fields will prevent an entire class of performance and consistency bugs.

Why PostgreSQL?

When choosing a relational database, the field narrows to MySQL, PostgreSQL, and SQL Server. PostgreSQL wins in most modern backend projects for five concrete reasons:

  1. Open source and free — No licensing cost. Companies can host it on their own servers without depending on a vendor. A huge number of startups and large companies choose Postgres for exactly this reason.
  2. Adheres strictly to the SQL standard — Any SQL query written for Postgres runs correctly on MySQL or SQL Server with minimal changes. Migrating to a different database system in the future is not a significant rewrite.
  3. Highly extensible — The official Postgres documentation is approximately 1,400 pages and covers practically every use case a SaaS product will ever encounter. It has a rich extension ecosystem for further customization.
  4. Proven reliability and scalability — Postgres is used in production by companies at massive scale and has decades of battle-testing behind it.
  5. Native JSON support — This is the most important differentiator. One reason developers choose MongoDB (non-relational) is the ability to store any arbitrary JSON. Postgres provides a native jsonb data type with full indexing and query support. You get relational data integrity and flexible JSON storage in the same database — eliminating the need to choose between them.

The common claim that MySQL has significant performance advantages over Postgres is only relevant at massive scale for very specific bottlenecks. Until you are serving millions of users and have identified a concrete bottleneck, Postgres is the correct default choice.

Data Types in PostgreSQL

Choosing the right data type for each column is one of the most important database design decisions. It directly affects data integrity, storage efficiency, and query performance.

Integer Types

Type Storage Use Case
smallint 2 bytes (−32,768 to 32,767) Small counters, lookup codes
integer 4 bytes General whole numbers
bigint 8 bytes (very large range) Row counts, large IDs in production
serial Auto-incrementing integer Simple auto-increment IDs
bigserial Auto-incrementing bigint Production primary keys (preferred)

Decimal / Floating Point Types

Type Accuracy Use Case
decimal(p, s) / numeric(p, s) Exact (no rounding) Price, currency, financial calculations
real Approximate (6 decimal digits) Scientific values where small drift is acceptable
double precision Approximate (15 decimal digits) Same as real, higher precision

Critical rule: For any field where accuracy matters — especially price or any monetary value — always use decimal or numeric. Floating point types store values in a binary format that can produce slightly different representations across different systems. For a value like area size (5.67891) where tiny discrepancies are acceptable, floating point is fine and also faster to process.

The decimal(10, 2) syntax means: maximum 10 total digits, with 2 digits after the decimal point. For example: 12345678.90.

String / Text Types

Type Behaviour Recommendation
char(n) Fixed length — pads with spaces if shorter Only use when all values always have identical length (e.g., day codes: "Mo", "Tu")
varchar(n) Variable length — max n chars Avoid; the "255" limit is a MySQL convention with no meaning in Postgres
text Unlimited length (up to ~250 MB) Always prefer this. No performance difference from varchar in Postgres.

The Postgres documentation itself recommends using text instead of varchar(255). The 255 limit is a habit carried over from MySQL, where it has a specific meaning. In Postgres, using varchar(255) is misleading — a new team member seeing it will assume the 255 has some intentional business meaning when it does not. Using text is cleaner, more honest, and eliminates the need for migrations later if you ever need to extend the "length limit."

Since backend engineers interact with the database through a driver (not raw SQL), enforce any character length constraints in the application layer instead, keeping migration files simpler.

Boolean

boolean — stores true or false. Standard and self-explanatory.

Date and Time Types

Type Stores Example
date Date only (no time) Due dates, birthdays
time Time only (HH:MM:SS) Daily schedule slots
timestamp Date + time Event timestamps without timezone
timestamptz Date + time + timezone Use this for created_at / updated_at.
interval Duration (e.g., "10 days") Subscription durations, retry windows

UUID

uuid — Universally Unique Identifier. A native type in Postgres. Highly preferred for primary keys in production systems because UUIDs are globally unique and unpredictable — unlike sequential integers which expose record counts and can be guessed. Postgres can auto-generate UUIDs using gen_random_uuid().

JSON Types

Type Storage Format When to Use
json Plain text (key-value string) Rarely; jsonb is almost always better
jsonb Binary (serialized by Postgres) Preferred. Faster queries, supports full indexing, richer operator support.

jsonb is a Postgres-specific type (not standard SQL) that converts JSON into a native binary format for faster processing and retrieval. This is what eliminates the need for a separate MongoDB instance to handle dynamic or unstructured data.

Arrays

Postgres allows you to store an array of any data type — integer[], text[], uuid[], jsonb[]. Useful for storing tags, permissions lists, or multi-valued fields without a join table.

Other Types

Postgres also supports inet (IP addresses), macaddr (MAC addresses), geometric types (points, lines), and xml — rarely needed by backend engineers but available.

Mental Model

Think of data types as column contracts. When you declare a column as decimal(10,2), you are making a promise to every future query: "the value in this field will always be a number with exactly 2 decimal places, no exceptions." The DBMS enforces this contract on every single insert and update — so your application never has to guard against invalid data arriving from the database.

Important Terms

  • UUID: Universally Unique Identifier — a 128-bit, globally unique value. Preferred over sequential integers for production primary keys.
  • jsonb: Binary JSON storage in Postgres. Faster to query than plain json, supports indexing.
  • decimal(p, s): Exact-precision number: p total digits, s after the decimal point.
  • timestamptz: Timestamp with timezone information — use for all audit fields.
  • text: Unlimited character storage. Postgres's recommended alternative to varchar.

Key Takeaways

  • Postgres wins for most projects: open source, SQL-standard, extensible, JSON support.
  • For monetary values: always decimal, never float.
  • For strings: always text, never varchar(255) (Postgres docs agree).
  • For primary keys: prefer uuid over serial integers in production.
  • For timestamps: use timestamptz (includes timezone).
  • For dynamic/unstructured data: use jsonb — eliminates the need for MongoDB.

Quick Revision

  • Postgres advantages: Open source, SQL-standard, extensible, reliable, native jsonb.
  • decimal vs float: Use decimal for prices; float for non-critical measurements.
  • char vs varchar vs text: char = fixed, varchar = limited, text = unlimited. Use text.
  • uuid: Globally unique, unpredictable, preferred for PKs.
  • jsonb: Binary JSON — indexable, faster than json type.
  • timestamptz: Always use this for created_at / updated_at.
