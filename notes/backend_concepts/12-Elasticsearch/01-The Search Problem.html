The Search Problem

Definition

The search problem refers to the difficulty of quickly and accurately finding relevant information within a massive dataset using traditional relational database queries.

Why do we need a better solution?

In a typical relational database setup, a basic text search is implemented using the ILIKE operator with wildcard percentage symbols (e.g., SELECT * FROM products WHERE name ILIKE '%laptop%'). While this works fine for a few thousand records, it fails completely at scale (millions of records) due to severe latency and a lack of relevance ranking.

The Librarian Analogy

Think of a relational database as a librarian who knows exactly where every book is located but has one fatal flaw when searching for a specific topic: they have to scan every single book one by one.

  • If you ask for books about "machine learning," the librarian examines the first book, the second book, and so on until a match is found.
  • In a library with millions of books, this sequential scanning process takes an unacceptable amount of time.

Disadvantages of Traditional Search

  • Latency at Scale: Scanning every row and performing character-by-character pattern matching is painfully slow (taking seconds instead of milliseconds).
  • No Sense of Relevance: A database cannot determine which results are more important. A book with the term in the title is treated the same as a book where the term is mentioned once on the last page.
  • Fragile Queries: Traditional queries break when users make typos. Searching for "laptpo" will yield zero results even if the intended query was "laptop."

Mental Model

Treat relational database text search like reading an entire dictionary from A to Z just to find a single definition. It's exhaustive but fundamentally unscalable for real-time applications.

Important Terms

Latency — The time it takes for a system to process a request and return a response.

ILIKE Operator — A SQL operator used for case-insensitive pattern matching.

Key Takeaways

  • Relational databases use sequential pattern matching for text searches.
  • This approach scales poorly as data volume grows into the millions.
  • Users expect search results in milliseconds, not seconds.

Quick Revision

  • Search using ILIKE '%term%' works for small datasets but fails at scale.
  • Traditional DBs lack relevance ranking.
  • Typo tolerance is practically impossible with standard SQL pattern matching.
