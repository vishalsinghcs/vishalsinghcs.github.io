Indexes and Triggers

Indexes make queries dramatically faster by replacing slow sequential scans with direct lookups. Triggers automate repetitive actions (like updating updated_at) that the database should manage on its own — without depending on the application code.

What is a Database Index?

Without an index, when you run a query like WHERE id = 'abc-123', the database performs a sequential scan — it reads every single row in the table one by one, comparing each row's ID to the target value, until a match is found.

Query: WHERE id = 'abc-123'
↓
Without Index: Scan row 1 → row 2 → row 3 → ... → row 1,000,000
↓
With Index: Look up 'abc-123' in lookup table → find disk location → fetch directly

An index is a separate data structure (a lookup table) maintained by the database. It stores a value from a specific column alongside a pointer to where that row lives on disk. When a query filters, joins, or sorts on an indexed column, the database can look up the target value in the index in a fraction of the time compared to scanning all rows.

Under the hood — B-tree: By default, Postgres implements indexes as a B-tree (Balanced Tree). This is important to understand: the index itself is not scanned linearly from top to bottom either. The B-tree is a sorted, self-balancing tree structure that allows the database to find any value using a process similar to binary search — splitting the search space in half at each step. For a table with 10 million rows, this means finding a value in roughly 24 comparisons (log₂ of 10,000,000 ≈ 23.25), rather than reading all 10 million rows. This is what gives indexes their dramatic speed advantage. Scanning the whole table is O(n); a B-tree lookup is O(log n).

Book analogy: An index in a database is exactly like the index section at the back of a textbook. Instead of reading all 500 pages to find "Chapter 4", you look up "Chapter 4" in the index and jump directly to page 67. The database's index works the same way — look up the value, get the disk location, retrieve the data directly.

When to Create an Index

The three primary conditions that signal a field should be indexed are:

Condition Example
Field is used in a WHERE clause WHERE status = 'pending' → index status
Field is used in a JOIN condition JOIN tasks ON tasks.project_id = projects.id → index project_id
Field is used in an ORDER BY clause ORDER BY created_at DESC → index created_at DESC

Note: Primary keys are automatically indexed by Postgres. You do not need to manually create an index on a primary key column. Foreign key columns, however, are not automatically indexed — you must add those yourself.

Index Order Matters for Sorting

An index can be created in ascending (ASC) or descending (DESC) order. If your most common query returns results sorted by created_at DESC (newest first), create the index in descending order. This allows Postgres to traverse the index in the same order as the query, without the overhead of sorting the result set afterward. 

-- Index for efficiently sorting users by newest first
CREATE INDEX idx_users_created_at_desc ON users (created_at DESC);

-- Index for JOIN lookups on tasks.project_id
CREATE INDEX idx_tasks_project_id ON tasks (project_id);

-- Index for filtering tasks by status (used in WHERE status = 'pending')
CREATE INDEX idx_tasks_status ON tasks (status);

The Cost of Indexes — Write Overhead

Indexes are not free. Every time a row is inserted, updated, or deleted, the database must also update all indexes on that table to reflect the change. More indexes = more work per write operation.

The trade-off is:

  • Reads become faster (index lookup instead of full scan)
  • Writes become slightly slower (index maintenance overhead)

For most applications where reads far outnumber writes, the trade-off strongly favors creating indexes on frequently queried fields. The decision guide:

  • Is this field used in a WHERE, JOIN, or ORDER BY condition?
  • Is the query containing this condition called frequently?
  • If yes to both → create the index.
  • If the query is rarely called, the overhead may not be worth it.
  • You can always delete an index later: DROP INDEX idx_name;

What is a Trigger?

A trigger is a stored database function that runs automatically when a specific event occurs on a table — such as an INSERT, UPDATE, or DELETE. Triggers allow you to automate actions at the database level without requiring the application code to remember to perform them.

The updated_at Problem

Every table has an updated_at column. Every time a row is updated, this column should reflect the current timestamp. Without automation, every UPDATE statement in the application must explicitly include updated_at = NOW(). This is error-prone — a developer refactoring a query might forget, leaving the updated_at stale.

A trigger solves this elegantly at the database level.

Creating the Trigger

Two steps: first create a trigger function, then attach it to each table as a trigger.

-- Step 1: Create a reusable trigger function
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();   -- Set updated_at to current timestamp
  RETURN NEW;               -- Return the modified row
END;
$$ LANGUAGE plpgsql;

-- Step 2: Attach the trigger to each table
CREATE TRIGGER set_updated_at_users
  BEFORE UPDATE ON users
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at_column();

CREATE TRIGGER set_updated_at_projects
  BEFORE UPDATE ON projects
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at_column();

-- Repeat for tasks, user_profiles, project_members, etc.

With this trigger in place, every UPDATE on the users table automatically sets updated_at to the current timestamp — regardless of whether the application code remembers to do it. The database is the single source of truth for this field.

How Triggers Work

  • BEFORE UPDATE — fires before the row is actually written to disk, allowing the function to modify the values being saved.
  • FOR EACH ROW — the trigger function is called once for each row being updated in a batch update operation.
  • NEW — a special variable inside trigger functions that holds the new values being written. Modifying NEW.updated_at changes what actually gets saved.

Down Migrations for Indexes and Triggers

When creating indexes and triggers in an up migration, the corresponding down migration must explicitly drop them in reverse order: 

-- Down migration
DROP TRIGGER IF EXISTS set_updated_at_users ON users;
DROP TRIGGER IF EXISTS set_updated_at_projects ON projects;
DROP FUNCTION IF EXISTS update_updated_at_column();

DROP INDEX IF EXISTS idx_users_created_at_desc;
DROP INDEX IF EXISTS idx_tasks_project_id;
DROP INDEX IF EXISTS idx_tasks_status;

Mental Model

An index is like a book's index section. Without it, finding "Chapter 4" means reading all 500 pages. With it, you look up "Chapter 4 → page 67" and jump directly there. A trigger is like a hotel's automatic door. You don't need to push it — the sensor (trigger condition) detects your approach (the UPDATE event) and opens the door (executes the function) automatically, every time, without any manual intervention.

Important Terms

  • Sequential Scan: Reading every row in a table one by one to find a match. Very slow at scale.
  • Index: A lookup data structure that maps a column's values to disk locations for fast retrieval.
  • Write Overhead: The additional work required to update all indexes when a row is inserted, updated, or deleted.
  • Trigger: A stored database function that fires automatically on specific table events (INSERT, UPDATE, DELETE).
  • NEW: A trigger special variable holding the new row values about to be written. Modifiable before saving.
  • BEFORE UPDATE: A trigger that fires before the row is written, allowing modification of values.

Key Takeaways

  • Without indexes, large table queries do sequential scans — extremely slow at scale.
  • Create indexes on fields used in WHERE clauses, JOIN conditions, and ORDER BY clauses.
  • Primary keys are auto-indexed. Foreign keys are NOT auto-indexed — add them manually.
  • Index order (ASC/DESC) matters for ORDER BY performance.
  • Indexes speed up reads but add write overhead — evaluate frequency before creating.
  • Triggers automate repetitive DB operations (like updating updated_at) at the DB level — more reliable than depending on application code.

Quick Revision

  • Sequential scan: No index → reads all rows. Slow at scale.
  • Index: Lookup table: value → disk location. Fast retrieval.
