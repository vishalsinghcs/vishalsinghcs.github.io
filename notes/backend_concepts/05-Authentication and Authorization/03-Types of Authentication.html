Types of Authentication

Understanding the four major types of authentication flows used by modern backend engineers.

1. Stateful Authentication

In stateful authentication, the server maintains an active record of all logged-in users in a fast persistent store (like Redis). The client simply holds a meaningless reference (the Session ID) in a cookie.

Use Cases: Ideal for standard web applications like banking portals or administrative dashboards where security is paramount, and administrators need the ability to instantly revoke a session or forcefully log a user out.

sequenceDiagram participant Client participant Server participant Redis Client->>Server: 1. Send Username/Password Server->>Server: 2. Validate Credentials Server->>Redis: 3. Generate & Store Session ID + User Data Server->>Client: 4. Send Session ID via Cookie Client->>Server: 5. Subsequent Requests (Includes Cookie) Server->>Redis: 6. Lookup Session ID Server->>Client: 7. Allow API Access

2. Stateless Authentication

In stateless authentication, the server stores nothing about the login session. It generates a signed JWT and hands it completely over to the client. The server relies solely on its secret key to verify the token on subsequent requests.

Use Cases: Perfect for highly scalable, distributed microservice architectures where routing every request through a centralized session database would cause a major bottleneck. Also very common for mobile application APIs.

sequenceDiagram participant Client participant Server Client->>Server: 1. Send Username/Password Server->>Server: 2. Validate & Sign JWT using Secret Key Server->>Client: 3. Send JWT to Client Client->>Server: 4. Subsequent Requests (Sends JWT in Header) Server->>Server: 5. Verify Signature using Secret Key Server->>Client: 6. Allow API Access

3. API Key Authentication

API Keys are used almost exclusively for machine-to-machine communication. Unlike humans using web browsers, a server communicating with another server (e.g., your backend calling OpenAI's chat API) cannot fill out a login form or interact with a UI.

Instead, a developer generates a cryptographically safe string (an API key) from the platform's UI, saves it on their own server, and attaches it programmatically to every request.

Use Cases: Used when a backend server needs to communicate with another service behind the scenes, such as calling Stripe's API to process a payment, or calling OpenAI's API to generate text.

4. OAuth 2.0 and OpenID Connect (OIDC)

Historically, if a new travel app wanted to read your Gmail to find flight tickets, it had to ask for your Google password. This was a massive security flaw—it gave the travel app total access to your account.

OAuth 2.0 solved this "Delegation Problem" by sharing Tokens instead of passwords. It allows you to grant specific, limited permissions (e.g., "Read Emails only") to a third-party app. A standard OAuth workflow involves four key components:

  • Resource Owner: The user who owns the data (You).
  • Client: The app requesting access (The travel app).
  • Resource Server: The server holding the data (Google's Gmail server).
  • Authorization Server: The server that authenticates the user and issues the token (Google's Auth server).
sequenceDiagram participant User as Resource Owner participant Client as Client (Travel App) participant Auth as Authorization Server participant Resource as Resource Server Client->>Auth: 1. Redirect user to Auth Server Auth->>User: 2. Prompt for login & consent User->>Auth: 3. Authenticate & Grant "Read Email" Auth->>Client: 4. Issue Access Token (Bearer Token) Client->>Resource: 5. Request emails + Access Token Resource->>Client: 6. Validate token & Return emails

To support different types of applications, OAuth 2.0 introduced various Flows (e.g., Authorization Code Flow for backend servers, and Client Credentials for M2M).

While OAuth solved authorization (access), OpenID Connect (OIDC) was built on top of it to solve authentication (identity). It introduced the ID Token (a JWT), allowing platforms to securely implement "Sign in with Google" by passing your profile data.

Mental Model

OAuth 2.0 is like giving a valet key to a parking attendant. The key can start the car and drive it a short distance (specific permission), but it cannot open the trunk or glovebox (limited access), and you never gave them your master house keys (password).

Important Terms

  • Machine-to-Machine (M2M): Servers communicating with other servers without human interaction.
  • Delegation Problem: The challenge of securely allowing a third-party application to access a user's resources on another server.
  • OIDC: OpenID Connect; adds identity verification on top of OAuth 2.0.

Key Takeaways

  • Stateful auth is ideal for most standard web applications because it allows instant session revocation.
  • Stateless auth is ideal for highly scalable, distributed microservice architectures.
  • API Keys are for servers; OAuth is for delegating permissions to third parties.

Quick Revision

  • Stateful: Requires a database/Redis lookup per request.
  • Stateless: Uses self-verifying JWTs.
  • API Keys: Used for M2M (e.g., calling OpenAI APIs).
  • OAuth 2.0: Solves delegation; grants scoped access without sharing passwords.
  • OIDC: Solves identity sharing (e.g., "Sign in with Google").
