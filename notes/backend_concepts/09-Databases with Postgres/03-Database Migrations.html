Database Migrations and Seeding

In production systems, you never apply database changes by manually typing SQL into a GUI tool. Migrations are the industry-standard mechanism for versioned, trackable, and reversible database schema changes.

Why Not Just Use a GUI?

Opening a tool like TablePlus and running SQL statements directly works fine for quick experimentation, but it creates serious problems in team and production environments:

  • There is no record of what changes were made, by whom, or when.
  • Other developers cannot reproduce the same database state on their local machines.
  • CI/CD pipelines and staging environments cannot be automated.
  • Rolling back a failed change is manual and error-prone.

Migrations solve all of these problems systematically.

What Are Database Migrations?

Migrations are a set of SQL files, stored in version control (Git), that represent every change ever made to your database schema — in the order those changes were applied.

db/migrations/
↓
001_create_users_table.sql
↓
002_create_projects_table.sql
↓
003_add_indexes_and_triggers.sql

Each file is named with a sequential number or a timestamp (timestamp naming is more common in teams because timestamps guarantee global uniqueness). A migration CLI tool reads these files in order and applies the SQL statements to the target database.

Up Migrations and Down Migrations

Every migration file typically contains two sections:

Section Purpose Example Content
UP Apply the change (move forward) CREATE TABLE users (...)
DOWN Reverse the change (rollback) DROP TABLE users

Down migrations exist so that if something goes wrong in production after a migration is applied, you can roll back to the previous database state cleanly — without manually reconstructing what the schema looked like before.

The down migration is the exact inverse of the up migration. If the up migration creates three tables and two types, the down migration must drop those three tables and then drop those two types — in reverse order to respect dependencies. 

-- migrate:up
CREATE TYPE project_status AS ENUM ('active', 'completed', 'archived');

CREATE TABLE projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  status project_status NOT NULL DEFAULT 'active',
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- migrate:down
DROP TABLE projects;
DROP TYPE project_status;

The Migration Tool: dbmate

dbmate (and alternatives like golang-migrate) is a CLI tool that manages migration files and tracks which migrations have been applied. You configure it by pointing it at a DATABASE_URL environment variable.

Command What it Does
dbmate new create_users_table Creates a new timestamped migration file
dbmate up Applies all pending up migrations in sequence
dbmate down Rolls back the most recently applied migration

How the Migration Tool Tracks State

When you first run a migration tool on a database, it creates a special metadata table — typically named schema_migrations. This table stores the version (or filename) of every migration that has been applied.

The next time you run dbmate up, the tool checks the schema_migrations table to determine which migrations have already been applied and skips them, only running the new ones. This prevents the same SQL statements from being executed twice (which would cause errors like "table already exists").

schema_migrations table
↓
Stores: applied migration versions
↓
dbmate up → checks table → skips already-applied files
↓
Applies only new migration files → updates schema_migrations

Seeding

Seeding is the process of populating a database with test data for development and testing purposes. After your schema migrations have created all the tables, you run seed migrations to insert fake-but-realistic data so you have something to actually test with.

Best practices for seeding:

  • Keep seed data in a separate migration file — don't mix it with schema creation statements.
  • Seed data only ever runs in development/staging environments — never in production.
  • Seeds often use CTEs (Common Table Expressions) to insert related data across multiple tables in a single readable SQL statement.
-- Example: Insert test users, then their profiles in one transaction using CTEs
WITH inserted_users AS (
  INSERT INTO users (email, full_name, password_hash)
  VALUES
    ('alice@example.com', 'Alice Brown', 'hashed_pw_1'),
    ('bob@example.com',   'Bob Johnson', 'hashed_pw_2')
  RETURNING id, email
)
INSERT INTO user_profiles (user_id, bio, phone)
SELECT
  id,
  CASE WHEN email = 'alice@example.com' THEN 'Product designer' ELSE 'Backend engineer' END,
  '555-0100'
FROM inserted_users;

Key Benefits of Using Migrations

  • Version control for your database — Every schema change is committed to Git alongside the application code that requires it. When you check out a branch, you can replay migrations to match exactly that branch's expected schema.
  • Team collaboration — Multiple developers can apply the same migrations to get identical database schemas locally without any manual coordination.
  • Rollback capability — If a migration causes a production outage, the down migration can revert the database schema quickly and safely.
  • Auditability — You can trace the entire history of your database schema by reading migration files in order.

Mental Model

Migrations are like Git commits for your database schema. Just as Git tracks every change to your code and lets you revert, branch, or replay history, migrations track every change to your database structure. The migration tool is like git apply — it reads a sequence of diff files and applies them one by one to bring the database up to the desired state.

Important Terms

  • Migration: A SQL file containing schema changes (CREATE TABLE, CREATE INDEX, etc.) to be applied to a database in a controlled, versioned way.
  • Up migration: The forward change — creates new tables, columns, or indexes.
  • Down migration: The reverse — drops what the up migration created, enabling rollback.
  • schema_migrations: The metadata table created by the migration tool to track which migrations have been applied.
  • Seeding: Inserting test/demo data into a development database after the schema is created.
  • CTE (Common Table Expression): A named temporary result set within a SQL query, used with WITH ... AS (...).
  • dbmate: A popular CLI migration tool for Postgres and other databases.

Key Takeaways

  • Never apply database changes manually in a GUI for production systems — always use migrations.
  • Every migration has an UP (apply) and a DOWN (rollback) section.
  • Migration tools track state via a schema_migrations table — preventing duplicate execution.
  • Seeding = test data for development only; keep it in a separate migration file from schema creation.
  • Commit migration files to Git alongside the code changes that require them.

Quick Revision

  • Migration: Versioned SQL file for schema changes.
  • Up: Apply change. Down: Reverse change.
  • dbmate new name → creates file. dbmate up → applies. dbmate down → rolls back.
  • schema_migrations: Tracks applied versions, prevents double-run.
  • Seeding: Inserting test data in development — separate migration file.
  • Benefits: Version control, team sync, rollback, auditability.
