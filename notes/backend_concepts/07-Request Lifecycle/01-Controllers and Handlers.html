Controllers and Handlers

The entry point of every API call — the layer responsible for parsing requests, binding data, and orchestrating the response.

Where Do Controllers Fit?

When a client sends an HTTP request, it reaches the server's entry point. The routing algorithm matches the URL and method, then passes the request to the associated controller (also called a handler). These terms are used interchangeably in most frameworks.

The controller is the first piece of code your request-specific logic touches. It sits between the raw HTTP world and your business logic.

The Two Default Objects

Every handler/controller always receives (at minimum) two objects from the runtime, regardless of language or framework:

  • Request Object (req): Contains all incoming data — path params, query params, request body, headers, and cookies.
  • Response Object (res): Used to send data back to the client — set status codes, headers, and the response body.

Responsibilities of a Controller

The controller's responsibilities are strictly about data input/output and HTTP logic. It should not run business logic itself. The standard flow inside a controller is:

  1. Bind / Deserialize: Extract and convert incoming data (JSON body, query params, path params) into the server's native format. In Go this means binding to a struct; in Python, a dictionary or class. In Node.js (Express), a middleware like express.json() usually does this automatically upstream.
    Failure result: return 400 Bad Request immediately.
  2. Validate and Transform: Run the validation and transformation pipeline against the bound data to ensure it meets the API's requirements before proceeding.
    Failure result: return 400 Bad Request with a descriptive error.
  3. Call the Service Layer: Pass the validated, clean data down to the service method to execute business logic.
  4. Format and Send Response: Based on the success or failure returned by the service, select the appropriate HTTP status code (200, 201, 204, 400, 500) and send the response body.

Why Separate the Controller from Business Logic?

The controller is intentionally kept "thin" — its job is to deal with the HTTP layer, not compute results. Business logic belongs in the Service layer. This separation provides:

  • Testability: You can unit test service methods without simulating HTTP requests.
  • Reusability: The same service method can be called from a REST controller, a CLI command, or a WebSocket handler.
  • Clarity: Looking at a service method, you should not be able to tell it is used in an HTTP server. It should read like plain logic.

Mental Model

Think of the Controller as a hotel receptionist. They don't cook the food or clean the rooms (that's the service and repository). They greet the guest (parse the request), check if they have a reservation (validate), notify the right department (call the service), and then tell the guest their room number (send the response).

Important Terms

  • Handler: A function that processes an HTTP request. "Handler" and "Controller" are used interchangeably.
  • Binding: The process of deserializing an incoming JSON body into a language-native data structure (struct, class, dict).
  • Request Object: Contains all incoming data from the client.
  • Response Object: Used to set status codes and send data back to the client.

Key Takeaways

  • Controllers handle HTTP-specific logic only — parsing, validating, responding.
  • Business logic belongs in the Service layer, not the Controller.
  • A controller that is hard to read likely contains too much business logic.
  • If binding or validation fails, the controller returns 400 immediately without calling the service.

Quick Revision

  • Step 1: Bind and deserialize incoming data.
  • Step 2: Validate and transform data.
  • Step 3: Call service layer.
  • Step 4: Send HTTP response (200/400/500).
  • Rule: Keep controllers thin — no business logic.
