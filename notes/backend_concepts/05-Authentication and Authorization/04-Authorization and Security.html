Authorization and Security Considerations

Once a user proves who they are (Authentication), the server must enforce rules on what they are allowed to do (Authorization).

Role-Based Access Control (RBAC)

In most modern applications, providing specific permissions to specific users is handled via RBAC. Instead of assigning individual permissions (like "can delete post") to a user directly, permissions are assigned to a Role (like "Admin"), and the Role is assigned to the user.

During a request, the server identifies the user's role (either stored in the session or inside the JWT payload) and checks it against the required permissions of the endpoint. If the user doesn't have sufficient privileges, the server returns a 403 Forbidden HTTP status.

Security: Generic Error Messages

When a user fails to log in, it is tempting to send helpful error messages like "User not found" or "Incorrect password". However, doing this gives attackers crucial information.

If an attacker sees "Incorrect password", they instantly know the email exists in your system. This allows them to build a verified list of target emails and brute-force the passwords. Always use generic error messages (e.g., "Authentication failed" or "Invalid credentials") regardless of whether the email or password was wrong.

Security: Timing Attacks

Even if you use generic error messages, attackers can sometimes figure out if an email exists based on how long the server takes to respond.

Consider the typical login workflow:

  1. Find the user by email in the database.
  2. If found, cryptographically hash the provided password and compare it to the stored hash.

Hashing a password is computationally slow by design. If an email does not exist, the server skips step 2 and responds instantly. If the email exists but the password is wrong, the server performs step 2 and the response is slightly delayed (e.g., +200ms). Attackers measure this delay to enumerate valid emails.

Defense: Use constant-time comparison algorithms or simulate a consistent fake delay so that failed logins always take the exact same amount of time, masking where the failure occurred.

Mental Model

Think of a Timing Attack like knocking on doors. If a house is empty, you hear silence immediately. If someone is home but ignores you, you might hear footsteps or a dog bark before silence. Attackers listen for the "footsteps" (server delay) to know a user exists.

Important Terms

  • RBAC: Role-Based Access Control; managing permissions via grouped roles rather than individually.
  • 403 Forbidden: The HTTP status code indicating successful authentication but failed authorization.
  • Timing Attack: Exploiting the difference in server response times to deduce hidden information.

Key Takeaways

  • Never tell a user if their email or their password was the cause of a login failure.
  • Authentication failure logic must execute in constant time to prevent enumeration attacks.

Quick Revision

  • Authorization answers "What can you do?"
  • Most systems use RBAC (e.g., User vs. Admin).
  • Missing permissions result in a 403 Forbidden.
  • Use Generic Error Messages to prevent email enumeration.
  • Use Constant Time Operations to prevent timing attacks.
