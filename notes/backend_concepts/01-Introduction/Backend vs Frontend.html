Backend vs Frontend

The fundamental difference lies in the runtime environment: frontends execute code in the user's browser, while backends execute code on a controlled, centralized server.

How Frontends Work

Unlike backends which process logic internally and only return results, frontends download code over the network and execute it locally on the client's device.

flowchart LR A[1. Document Fetch] --> B[2. Resource Fetch] B --> C[3. Painting CSS] C --> D[4. Hydration JS]
  1. Document Fetch: The browser makes an HTTP request and fetches the initial HTML document.
  2. Resource Fetch: The browser parses the HTML and initiates separate requests for linked resources like CSS, JavaScript, fonts, and images.
  3. Painting: The browser applies the fetched CSS rules to paint the visual layout of the page.
  4. Hydration: The fetched JavaScript is executed, attaching event listeners to buttons and forms to make the static page interactive.

Why can't we write backend logic in the frontend?

Because the frontend runs on unpredictable, untrusted client devices, it faces severe limitations that make centralized backend processing necessary.

Limitation Description
Security & Sandboxing Browsers run code in isolated sandboxes to protect user devices. Frontend code cannot access the file system, read environment variables, or safely hold secret API keys.
CORS Policies Browsers restrict external API calls (Cross-Origin Resource Sharing) to prevent malicious scripts from stealing data. You cannot freely connect to any external server from frontend code.
Database Connections Databases require persistent socket connections and binary data parsing. Browsers cannot maintain these connections efficiently. Furthermore, millions of clients connecting directly would instantly overwhelm the database.
Computing Power Client devices range from powerful desktops to incredibly slow smartphones. Heavy business logic running on the frontend leads to lag, crashes, and severe battery drain.

Comparison

Feature Frontend Backend
Runtime Client's Browser Centralized Server
Security Untrusted & Sandboxed Trusted & Controlled

Mental Model

Think of a restaurant. The Frontend is the menu and the waiter taking your order at the table. The Backend is the centralized kitchen that has access to all ingredients (data), securely prepares the meal, and manages multiple orders simultaneously.

Important Terms

  • Hydration — The process of executing JavaScript to attach event listeners to a static HTML page, making it interactive.
  • Connection Pool — A cache of active database connections maintained by a backend server to efficiently handle high traffic without constantly opening/closing connections.
  • Sandbox — A restricted execution environment enforced by browsers that prevents code from accessing the host operating system.
  • CORS — A browser security policy that restricts web pages from making requests to a different domain than the one that served the web page.

Key Takeaways

  • Frontends download logic and run it locally; Backends execute logic internally and send results.
  • Browsers enforce strict security, preventing direct file system access and cross-origin network calls.
  • Backends manage connection pools to protect databases from being overwhelmed by too many direct client connections.

Quick Revision

  • Frontend = Browser runtime.
  • Backend = Server runtime.
  • Browsers fetch HTML, then fetch resources, paint CSS, and hydrate JS.
  • Frontend logic cannot securely hold secrets or API keys.
  • CORS prevents arbitrary cross-domain API calls.
  • Backends have dedicated, scalable compute resources to handle heavy logic.
