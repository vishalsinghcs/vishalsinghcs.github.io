HTTP Methods and CORS

HTTP methods define the semantic intent of a request. Instead of every request doing the same structural thing, methods provide a standardized vocabulary.

Common HTTP Methods

  • GET: Used to fetch data. Should not modify any server state.
  • POST: Used to create new data on the server. Includes a request body.
  • PATCH: Used to update existing data selectively (like an append or a partial update).
  • PUT: Used to completely replace a resource. The data in the body fully overwrites the previous instance. (Rule of thumb: use PATCH unless a full replacement is required).
  • DELETE: Used to delete a resource from the server.
  • OPTIONS: Used primarily for CORS pre-flight to fetch server capabilities.

Idempotent vs Non-Idempotent

Idempotency refers to whether making the same request multiple times produces the exact same result as making it once.

Idempotent Methods (GET, PUT, DELETE)

You can execute these methods repeatedly, and the end state on the server remains the same.

  • GET: Fetching data doesn't change it.
  • PUT: Overwriting the exact same resource with the exact same data multiple times results in the same state.
  • DELETE: Deleting a resource leaves it deleted regardless of how many times you subsequently try to delete it.

Non-Idempotent Methods (POST)

Calling these methods multiple times yields different results. For example, submitting a POST request to create a user three times will result in three distinct users being created.

CORS (Cross-Origin Resource Sharing)

Browsers enforce a Same-Origin Policy which restricts web pages from making requests to a different domain. CORS is a security mechanism that allows servers to explicitly specify who can access their resources across domains.

Simple Requests

A request is considered "simple" if it uses GET, POST, or HEAD, and only utilizes basic headers and standard content types (like application/x-www-form-urlencoded or multipart/form-data).

Client sends GET to api.example.com (Origin: example.com)
↓
Server verifies Origin, responds with Access-Control-Allow-Origin: example.com
↓
Browser validates header and lets response pass

If the server doesn't respond with a matching Access-Control-Allow-Origin, the browser throws a CORS error and blocks the response.

Pre-Flight Requests

If a request uses methods like PUT or DELETE, includes custom/non-simple headers (like Authorization), or uses a content-type like application/json, the browser fires a "pre-flight" request using the OPTIONS method.

Browser auto-sends OPTIONS (inquiring about methods/headers allowed)
↓
Server responds 204 (No Content) with Access-Control-Allow-Methods, Headers, Max-Age
↓
Browser validates allowed capabilities
↓
Browser sends the original request (e.g., PUT)

The Access-Control-Max-Age header tells the browser to cache this pre-flight result (e.g., for 24 hours), saving bandwidth for future requests.

Important Terms

  • Origin: The combination of protocol, domain, and port. Even running apps on different localhost ports constitutes different origins.
  • Idempotency: The property of an operation that can be applied multiple times without changing the result beyond the initial application.

Quick Revision

  • GET, PUT, DELETE = Idempotent.
  • POST = Non-idempotent.
  • PATCH = Selective update; PUT = Complete replacement.
  • CORS prevents arbitrary cross-origin requests.
  • Simple requests just require Access-Control-Allow-Origin.
  • Pre-flight requests use OPTIONS to ask the server for permission before sending the real request.
