HTTP Basics

HTTP (Hypertext Transfer Protocol) is the primary medium through which browsers talk to our servers to send or receive data. It operates primarily at the Application Layer (Layer 7) of the OSI model.

Core Principles of HTTP

There are two foundational ideas at the heart of the HTTP protocol:

1. Statelessness

HTTP has no memory of past interactions. Each request is an isolated event and must contain all the information necessary (like authentication tokens or session info) for the server to process it.

  • Simplicity: Server architecture is simplified as it doesn't need to store session information.
  • Scalability: It's easy to distribute requests across multiple servers since no single server needs to track a session.
  • State Management: Since it is inherently stateless, developers often implement techniques like cookies, sessions, or tokens to maintain continuity (e.g., user logins).

2. Client-Server Model

A typical HTTP request flow always involves a client and a server. Communication is always initiated by the client.

  • Client: Typically a web browser or an application that initiates communication by sending a request (providing URLs, headers, etc.).
  • Server: Hosts resources and waits for incoming requests. It processes the client's request and sends back a response (HTML, JSON, etc.).

Evolution of HTTP

Over the years, HTTP has evolved to become faster and more efficient:

  • HTTP 1.0: Each request opened a new connection. This led to inefficiencies because establishing and closing TCP connections for every request/response slowed down performance.
  • HTTP 1.1: Introduced persistent connections, allowing multiple requests and responses over the same TCP connection. Added chunked transfer encoding and better caching.
  • HTTP 2.0: Introduced multiplexing, allowing multiple concurrent requests/responses over a single connection. Uses binary framing instead of text, and supports header compression and server push.
  • HTTP 3.0: Built on QUIC (a transport layer protocol over UDP rather than TCP). It improves performance with faster connection establishment, reduced latency, and better handling of packet loss without head-of-line blocking.

HTTP Messages

Communication occurs through messages, which are categorized into requests and responses.

Request Message (Client to Server)

The message the client uses to ask the server to do something.

  • Request Method: e.g., GET, POST.
  • Resource URL: The path to the requested resource.
  • HTTP Version: e.g., HTTP/1.1.
  • Headers: Key-value pairs containing metadata (e.g., Host).
  • Blank Line: Signifies headers are over and the body begins.
  • Request Body: Optional payload of data sent to the server.

Response Message (Server to Client)

The message the server sends back in answer to a client's request.

  • HTTP Version: e.g., HTTP/1.1.
  • Status Code: A number indicating the outcome (e.g., 200).
  • Status Value: Text description of the outcome (e.g., OK).
  • Response Headers: Metadata about the response.
  • Blank Line: Signifies headers are over.
  • Response Body: The requested resource (HTML, JSON, text).

Mental Model

Think of HTTP as an ongoing conversation where one person (the server) has amnesia. Every time the client speaks to the server, they must re-introduce themselves and provide all context for the request.

Important Terms

  • TCP: Transmission Control Protocol. The reliable, connection-based protocol HTTP relies on for its transport.
  • HTTPS: A secure version of HTTP that uses TLS/SSL to encrypt data.
  • OSI Layer 7: The Application Layer where HTTP operates.

Quick Revision

  • HTTP is stateless and follows a client-server model.
  • Clients initiate communication; servers respond.
  • HTTP/1.1 brought persistent connections.
  • HTTP/3 uses QUIC over UDP instead of TCP.
  • Messages consist of a start-line, headers, a blank line, and an optional body.
