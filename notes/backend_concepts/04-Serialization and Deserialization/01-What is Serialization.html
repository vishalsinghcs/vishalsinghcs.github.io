What is Serialization?

Serialization is the process of converting complex data structures from a specific programming language into a universal format for network transmission.

The Language Barrier Problem

In modern web architecture, the client (frontend) and the server (backend) are often written in entirely different programming languages. They run in different environments and handle memory and types differently.

For example, imagine a frontend built with a highly dynamic, uncompiled language like JavaScript, communicating with a backend built in a strict, memory-safe, compiled language like Rust.

When JavaScript wants to send user data, it has a JavaScript Object in memory. If it sends this raw memory object over the internet, the Rust server will have absolutely no idea how to read it because Rust expects a strictly typed struct. They speak different languages.

The Solution: A Common Standard

To solve this, both machines agree on a common standard format for data transmission. This acts as a universal translator.

  • Serialization: The client takes its language-specific data (e.g., JS Object) and converts it into the agreed-upon common standard format string before sending it over the network.
  • Deserialization: The receiving server takes this standard string and reconstructs it back into its own language-specific data type (e.g., Rust Struct) so it can perform business logic.

The OSI Model Mental Model

If you study networking via the OSI Model, you know that data transmission is incredibly complex. Data starts at the Application layer, breaks down into data frames, gets chunked into IP packets, hits the physical transmission layer, and turns into raw binary (1s and 0s) sent as voltage signals over optical fiber.

As a backend engineer building APIs, you do not need to manually handle IP packets or optical voltage.

Your mental model sits entirely at the Application Layer. You only need to ensure your application can convert data to the common standard format, hand it off to the underlying networking libraries, and know that it will arrive on the other side in that exact same standard format.

Mental Model

Think of a diplomat from Japan and a diplomat from Brazil trying to negotiate a treaty. Neither speaks the other's native language. So, they both agree to translate their documents into English (the common standard) before exchanging them. Once received, they translate the English documents back into their native tongues.

Important Terms

  • Serialization: Converting in-memory data objects into a standard transmittable string/byte format.
  • Deserialization: Reconstructing standard format strings/bytes back into native in-memory objects.
  • Application Layer: The top OSI layer where data formats (like JSON) are parsed before hitting the network stack.

Key Takeaways

  • Frontend and backend often speak completely different programming languages.
  • Directly sending native objects over the network is impossible due to memory format mismatches.
  • A universal format acts as a translator between these systems.

Quick Revision

  • Clients and servers often use completely different languages (e.g., JS vs Rust).
  • They cannot send native memory objects directly over the network.
  • Serialization: Converting native data to a common format.
  • Deserialization: Parsing a common format back into native data.
  • Backend devs operate at the Application layer and rarely manage raw network packets.
