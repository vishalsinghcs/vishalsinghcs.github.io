Schema Design and Constraints

Schema design is where you translate business requirements into tables, columns, and relationships. Constraints are the rules that protect those tables from bad data — enforced at the database level, not the application level.

Naming Conventions

Before writing any schema, establish consistent naming rules. In Postgres:

  • Table names: plural, snake_case (e.g., users, project_members)
  • Column names: lowercase, snake_case (e.g., full_name, created_at)
  • Never use camelCase — Postgres is case-insensitive by default. If you write fullName, Postgres silently converts it to fullname. To preserve case you must wrap identifiers in double quotes every time — which is tedious and error-prone in application code.

Constraints

A constraint is a rule applied to a column that Postgres enforces on every insert and update. Violating a constraint produces a database-level error, preventing bad data from ever reaching storage.

Constraint Effect Example
NOT NULL Column cannot contain NULL email TEXT NOT NULL
UNIQUE All values in the column must be distinct email TEXT NOT NULL UNIQUE
PRIMARY KEY NOT NULL + UNIQUE; uniquely identifies the row id UUID PRIMARY KEY
FOREIGN KEY / REFERENCES Value must match an existing value in another table's primary key owner_id UUID REFERENCES users
CHECK Custom boolean condition must be true for the row to be accepted CHECK (priority BETWEEN 1 AND 5)
DEFAULT Provides a fallback value when none is supplied on insert status project_status NOT NULL DEFAULT 'active'

Important rule: By default in Postgres, every column accepts NULL unless you explicitly write NOT NULL. For the vast majority of columns — more than 70% — you should add NOT NULL. Omitting it allows application bugs or automated scripts to silently insert NULL values, which corrupts the database over time.

Enums — Custom Data Types

When a field has a fixed, predefined set of allowed string values, use a custom ENUM type rather than a plain text column. This provides two benefits: data integrity (the database rejects any value not in the list) and documentation (anyone reading the migration immediately understands all valid states of the field). 

-- Creating custom enum types
CREATE TYPE project_status AS ENUM ('active', 'completed', 'archived');
CREATE TYPE task_status AS ENUM ('pending', 'in_progress', 'completed', 'cancelled');
CREATE TYPE member_role AS ENUM ('owner', 'admin', 'member');

-- Using a custom enum type on a column
status project_status NOT NULL DEFAULT 'active'

Without enums, enforcing "status must be one of these three values" must be done in application code. But application code changes constantly — a developer refactoring a handler might accidentally remove or break the validation, silently allowing invalid data into the table. The enum enforces this at the database level no matter what.

Relationships Between Tables

Relational databases represent relationships between entities through keys. There are three types of relationships:

One-to-One (1:1)

Each row in Table A corresponds to exactly one row in Table B, and vice versa. Example: users → user_profiles. A user has exactly one profile; a profile belongs to exactly one user.

Why separate tables? Storing all profile fields in the users table forces migrations to that table every time you add a new profile field. Separating them isolates concerns — the users table stays stable, while user_profiles can grow independently.

Implementation: The secondary table uses the foreign key of the primary table as its own primary key. 

CREATE TABLE user_profiles (
  user_id UUID PRIMARY KEY REFERENCES users,  -- PK + FK
  avatar_url TEXT,
  bio TEXT,
  phone TEXT
);

One-to-Many (1:M)

One row in Table A corresponds to many rows in Table B. Example: one project has many tasks. A task belongs to exactly one project.

Implementation: Add a foreign key column to the "many" table referencing the "one" table's primary key. 

CREATE TABLE tasks (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects ON DELETE CASCADE,
  title TEXT NOT NULL,
  ...
);

Many-to-Many (M:M)

Many rows in Table A relate to many rows in Table B. Example: a user can be part of many projects; a project can have many users. You cannot express this with a single foreign key in either table.

Implementation: Create a linking table (also called a join table or junction table) that holds foreign keys to both tables. The primary key of the linking table is a composite primary key — the combination of both foreign keys. 

CREATE TABLE project_members (
  project_id UUID NOT NULL REFERENCES projects ON DELETE CASCADE,
  user_id    UUID NOT NULL REFERENCES users    ON DELETE CASCADE,
  role       member_role NOT NULL DEFAULT 'member',
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (project_id, user_id)  -- Composite PK
);

The composite primary key guarantees that a user can only be added to the same project once — it enforces uniqueness of the combination of the two keys.

Referential Integrity — ON DELETE Behaviour

When a foreign key relationship exists and a parent row is deleted, the database must decide what to do with the child rows. This is configured with the ON DELETE clause:

Option Behaviour on Parent Delete When to Use
RESTRICT Fails with error — cannot delete parent if children exist When children must not be orphaned. E.g., cannot delete a user who owns projects.
CASCADE Automatically deletes all child rows When children are meaningless without the parent. E.g., deleting a project deletes all its tasks.
SET NULL Sets the FK column to NULL in child rows When the relationship is optional. E.g., deleting a user unassigns their tasks (sets assigned_to = NULL).
SET DEFAULT Sets the FK column to its DEFAULT value Rare — only when a meaningful default reassignment exists.

Mental Model

Think of constraints as bouncers at a nightclub door. The NOT NULL bouncer doesn't let anyone in without an ID. The UNIQUE bouncer won't let the same person in twice. The FOREIGN KEY bouncer checks your name against the VIP list (another table). The CHECK bouncer verifies your age (custom condition). They don't care which part of the application is sending data — the bouncer enforces the rules for everyone, always.

Important Terms

  • Primary Key: A column (or set of columns) that uniquely identifies every row. Implicitly NOT NULL and UNIQUE.
  • Foreign Key: A column that references the primary key of another table, enforcing the relationship.
  • Enum: A custom Postgres type with a fixed set of allowed string values.
  • Composite Primary Key: A primary key made of two or more columns combined — used in linking tables for M:M relationships.
  • Linking Table: An intermediary table used to model many-to-many relationships.
  • Referential Integrity: The guarantee that all foreign key values have corresponding rows in the referenced table.

Key Takeaways

  • Use snake_case for all table and column names in Postgres — camelCase causes silent case-folding bugs.
  • Add NOT NULL to 70%+ of your columns unless NULL is intentionally valid for that field.
  • Use ENUM types instead of plain text for fields with predefined allowed values — for integrity and documentation.
  • 1:1 → FK as PK in child table. 1:M → FK in the "many" table. M:M → linking table with composite PK.
  • ON DELETE RESTRICT = fail. CASCADE = delete children. SET NULL = null children. Choose deliberately.

Quick Revision

  • snake_case for all table/column names — never camelCase.
  • NOT NULL should be on most columns by default.
  • ENUM for fixed allowed values — enforced at DB, not app level.
  • 1:1 → child table uses parent PK as its own PK.
  • 1:M → FK in the "many" side table.
  • M:M → linking table with composite PK of both FKs.
  • RESTRICT: block delete. CASCADE: delete children. SET NULL: null children.
