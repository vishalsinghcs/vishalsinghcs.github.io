Types of Errors in Backend Systems

flowchart TD App[Backend App] App -->|Logic| LE[Logic Errors: Silent & Dangerous] App -->|Data| DE[(Database Errors: Deadlocks)] App -->|Network| EE[External APIs: Outages & 429s] App -->|Payload| VE[Validation: Bad Request Data] App -->|Startup| CE[Config Errors: Missing Variables]

In backend engineering, the question is not whether errors will happen, but how you handle them when they do. Developing a fault-tolerant mindset means being prepared for various failure scenarios.

1. Logic Errors

These are the sneakiest and most dangerous type. Your code runs without crashing, but the business logic is fundamentally wrong. For example, accidentally applying a discount twice resulting in negative shipping costs. These can go unnoticed for weeks, quietly corrupting data or causing financial loss.

2. Database Errors

Because backend applications rely heavily on data, database errors can bring down the entire system:

  • Connection Errors: The app cannot talk to the database (e.g., network down, connection pool exhausted).
  • Constraint Violations: Attempting to insert a duplicate email (Unique Constraint) or referencing a non-existent ID (Foreign Key Constraint).
  • Query Errors: Malformed SQL, accessing non-existent tables, or complex queries timing out.
  • Deadlocks: Multiple operations waiting on each other, creating a circular dependency.

3. External Service Errors

Modern applications depend on 3rd-party providers (Stripe, AWS S3, Auth0, Resend). Each is a potential point of failure:

  • Network & Outages: DNS failures, timeouts, or major cloud provider outages.
  • Rate Limiting (429): Hitting an API too frequently. Requires strategies like exponential backoff.
  • Authentication Errors: Expired tokens or invalid credentials when talking to external APIs.

4. Input Validation Errors

These occur when clients send bad data. The validation layer is the first line of defense. By catching malformed emails, out-of-range numbers, or missing required fields early, we can safely return a 400 Bad Request and prevent bad data from reaching core business logic.

5. Configuration Errors

Missing environment variables (e.g., an OpenAI API Key) can crash the application. It is always better to validate all required configurations at startup so the app crashes immediately during deployment, rather than failing randomly at runtime when real users trigger an API call.

Mental Model

Treat your application like a fortress. You must anticipate breaches (logic errors), supply line cuts (database/external outages), and bad actors (validation errors) before they actually occur.

Important Terms

Exponential Backoff: Waiting progressively longer periods (1m, 2m, 4m) before retrying a failed external API request.

Constraint Violation: Breaking database rules like uniqueness or foreign key references.

Key Takeaways

  • Logic errors are dangerous because they don't crash the app.
  • Fail fast: validate configuration variables on startup, not during runtime.

Quick Revision

  • Types: Logic, DB, External, Validation, Config.
  • Rate limits (429) require exponential backoff.
  • Validation is the first line of defense (400 Bad Request).
