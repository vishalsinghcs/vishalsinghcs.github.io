Storage and Sources

Where and how you store your configurations depends heavily on security, speed, and the environment your application is running in.

1. Environment Variables

The most ubiquitous way of passing configuration to applications across all modern languages (Node.js, Python, Go, etc.).

  • Local Development: Stored in `.env` files. Libraries like `dotenv` load these values into the operating system's environment on startup.
  • Cloud/Container Deployments: Cloud providers inject environment variables directly into the container running your application.

2. Configuration Files

For complex, hierarchical settings, simple key-value pairs aren't enough. Configuration files offer a structured format.

  • JSON: Structured but lacks support for comments, making it hard to document settings.
  • YAML: Highly preferred because it supports comments and clean hierarchical data (e.g., `config.yaml`). Commonly used in Go applications and open-source projects.
  • TOML: A newer standard gaining popularity for its explicit syntax and ease of reading.

3. Cloud Native & Key-Value Stores

Lightweight, distributed data stores specifically designed for configuration in distributed systems.

  • Tools: Consul, etcd.
  • Use Case: They act like centralized environment variables that can be updated dynamically across a cluster.

4. Cloud Secrets Managers

Dedicated services for securely storing sensitive configurations in a distributed infrastructure.

  • Providers: HashiCorp Vault, AWS Parameter Store, Azure Key Vault, Google Secret Manager.
  • Workflow: Deployment pipelines fetch encrypted secrets from these providers and inject them into the application's environment.

5. Hybrid Strategies

Large applications often use a combination of sources prioritized in a specific order. When the application starts, it builds a single configuration object.

Priority 1: AWS Parameter Store
↓
Priority 2: config.yaml
↓
Priority 3: Environment Variables (.env)

Mental Model

Think of configuration storage as an inheritance hierarchy. Your default `.yaml` file is the base class. Your environment variables override specific settings depending on the context, and your secrets manager injects the highly secure, hidden traits.

Important Terms

HashiCorp Vault: A popular cloud-agnostic tool used for managing secrets and protecting sensitive data across distributed systems.

Key Takeaways

  • Env vars are standard but lack structure.
  • YAML is preferred for structured configs due to comment support.
  • Cloud Secret Managers are essential for security.
  • Applications combine multiple sources to build final configs.

Quick Revision

  • Environment: `.env` loaded via OS context.
  • Files: YAML > JSON (because of comments).
  • Key-Value: Consul, etcd.
  • Cloud Secrets: Vault, Parameter Store.
  • Hybrid: Priority-based loading.
