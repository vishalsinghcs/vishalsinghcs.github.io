Custom Actions and Best Practices

Not every operation fits neatly into CRUD. Custom actions handle complex server-side operations that go beyond a simple database write. Best practices ensure your API remains intuitive, consistent, and pleasant to integrate — long after you've written it.

What is a Custom Action?

A custom action is any operation that:

  • Is not simply creating a new resource
  • Is not a fetch, update, or delete operation
  • Triggers a complex, multi-step process on the server side

Example 1 — Archive an Organization: At first glance, archiving looks like updating the status field to "archived". But in reality, archiving an organization might need to:

  • Delete or deactivate all projects under that organization
  • Cancel all active tasks under those projects
  • Send notification emails to all members
  • Revoke API keys associated with the organization

A PATCH /organizations/:id with {"status": "archived"} cannot express this richness. Archiving is a custom action.

Example 2 — Clone a Project: Cloning a project involves creating a new project, copying all its tasks, potentially cloning sub-resources, and maybe sending an email to the project owner. A standard POST /projects with a copied payload doesn't capture the "clone" semantics, and doesn't trigger the server-side logic that makes cloning meaningful.

Custom Action URL Pattern

All custom actions follow this URL structure:

POST /resource-name/:id/action-name
Segment Example Meaning
Resource (plural) /organizations The resource type this action applies to
Dynamic parameter /:id Which specific resource to act on
Action name /archive The name of the action — a verb that describes what happens

Full examples:

  • POST /organizations/abc123/archive
  • POST /projects/xyz789/clone
  • POST /users/me/send-verification-email

Response Codes for Custom Actions

Do not blindly assume a POST always returns 201. The response code for a custom action depends on what the server actually does:

Server Action Response Code Example
No new resource created 200 POST .../archive — updates status, no new resource
A new resource created 201 POST .../clone — creates a new cloned project

API Design Best Practices

The lecture concludes with a definitive list of principles for building APIs that people enjoy using:

1. Provide Interactive Documentation (Swagger / OpenAPI)

Before your API goes into production, integrate an OpenAPI-compatible documentation tool (Swagger UI is the most popular). This gives consumers an interactive playground where they can discover, read about, and test your endpoints without reading source code. Maintaining your Swagger documentation up to date is one of the most distinguishing marks of a senior backend engineer.

2. Be Consistent Across All Resources

Every decision you make about your API design becomes a pattern. If you use description as a field name in one endpoint, use description everywhere. If you use plural URLs for organizations, use plural URLs for projects. Never rename the same concept differently across resources. Inconsistency forces API consumers to read your docs for every single endpoint instead of making safe assumptions from the first one they integrated.

3. Provide Sensible Defaults

Never require a client to send information that has an obvious default. Common examples:

  • If page is not sent → default to 1
  • If limit is not sent → default to 10
  • If a new organization is created without a status → default to "active"
  • If sortOrder is not sent → default to desc

The goal is to make the most common use case require the least amount of effort from the client.

4. Never Use Abbreviations

The person who created the API has full context about what DSC or orgId means. The person integrating it does not. Always use full, readable names: description, organizationId. Abbreviations are a false shortcut that cause confusion and bugs downstream.

5. Design the Interface Before Writing Code

Your API interface is a contract. Once clients depend on it, changing it breaks them. Designing the interface first — using tools like Insomnia or Postman — forces you to think from the consumer's perspective rather than the implementation perspective. Decisions made at the interface level are far cheaper than refactoring code after the fact.

Summary: REST API Design Cheat Sheet

Rule Do This Not This

Mental Model

A custom action is like a vending machine button that dispenses coffee — it doesn't just "add coffee" (CRUD), it triggers a whole chain: heat water, grind beans, extract espresso, steam milk. You push one button (POST .../make-coffee) and the machine handles the complexity. From the outside, it is a single action. On the inside, it's a coordinated sequence.

Important Terms

  • Custom Action: A verb operation that doesn't fit CRUD, expressed as POST /resource/:id/action.
  • OpenAPI / Swagger: Standard for describing REST API interfaces. Enables interactive documentation.
  • Sensible Default: A server-provided fallback value for an optional parameter (e.g., page=1 if not sent).
  • API Contract: The agreed-upon interface between producer (server) and consumer (client). Changing it breaks existing consumers.

Key Takeaways

  • Custom actions use POST /resource/:id/action-name — when the operation triggers complex server-side logic beyond a simple update.
  • Custom action response codes depend on what the server does — not always 201.
  • The most important practices: consistency, sensible defaults, no abbreviations, and maintaining Swagger docs.
  • Design the interface first with API clients (Insomnia, Postman) — before writing any server code.

Quick Revision

  • Custom action pattern: POST /resource/:id/verb
  • Archive (no new resource): → 200
  • Clone (new resource created): → 201
  • Swagger: Interactive docs, must stay updated.
  • No abbreviations: description not DSC.
  • Defaults: Server, not client, is responsible.
