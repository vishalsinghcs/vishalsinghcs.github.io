Transformations

Converting and normalizing incoming data into desirable formats before or after validation occurs.

What is Transformation?

While validation checks if data is correct, transformation modifies the data. This is usually done in the same pipeline as validation (The Validation and Transformation Pipeline) so that all data preparation logic stays in one unified place.

1. Type Casting

Type casting is the act of forcing one data type to convert into another data type. This is especially critical when dealing with HTTP components that do not natively support strict data types.

  • Query Parameters: When a client sends a GET request to /api/bookmarks?page=2&limit=20, the values 2 and 20 are transmitted as Strings by the HTTP protocol.
  • The Problem: If your validation pipeline strictly requires page and limit to be Numbers, it will fail and reject the request.
  • The Solution: A transformation step intercepts the payload, recognizes that the fields are meant to be numeric, casts the strings "2" and "20" into integers 2 and 20, and then passes them to the validation layer.

2. Data Normalization

Normalization involves formatting string data consistently to prevent business logic errors later in the service layer.

  • Emails: A user might input their email as User.Name@Example.com. A transformation step will automatically convert this to lowercase: user.name@example.com, preventing duplicate account creations caused by case sensitivity in the database.
  • Phone Numbers: The client sends a phone number like 1234567890. The transformation pipeline detects this and prepends the required + character (e.g., +1234567890) so that the downstream SMS service layer does not crash.

The Pipeline Flow

In practice, transformation and validation run sequentially. Often, data is transformed first (e.g., casting a string to a number) so that validation can run successfully against it.

flowchart LR Input([Raw Client Data]) --> T1[Transform: Type Cast] T1 --> V1[Validate: Type Check] V1 --> T2[Transform: Normalize] T2 --> V2[Validate: Semantic Check] V2 --> Output([Clean, Safe Data])

Mental Model

Validation is the inspector; Transformation is the tailor. Validation checks if the suit fits. Transformation alters the suit so that it fits perfectly before the inspector even looks at it.

Important Terms

  • Type Casting: Forcing data to convert from one primitive type to another (e.g., String to Number).
  • Normalization: Standardizing data into a consistent format (e.g., lowercasing all emails).

Key Takeaways

  • Query parameters and path parameters are always Strings by default in HTTP. They must be cast to numbers or booleans if the API requires them to be.
  • Group validations and transformations together in a single pipeline so developers only have to look in one place to see data rules.

Quick Revision

  • Validation: Confirms data matches requirements.
  • Transformation: Modifies data to match requirements.
  • Casting Example: "5" -> 5.
  • Normalization Example: "TEST@ME.com" -> "test@me.com".
