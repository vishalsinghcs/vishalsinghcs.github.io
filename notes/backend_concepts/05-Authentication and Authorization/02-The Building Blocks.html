The Building Blocks of Modern Authentication

To understand modern authentication flows, we must first understand the three core components that make them possible: Sessions, JWTs, and Cookies.

1. Sessions (Stateful Architecture)

By design, HTTP is a stateless protocol. It treats every request as an isolated interaction with no memory of past exchanges. While this was fine for early static websites, modern dynamic applications (like e-commerce sites needing carts, or social networks keeping users logged in) require memory.

Sessions were introduced to establish a temporary server-side context for each user. When a user logs in, the server generates a unique session ID, stores the user's data in a persistent store (like Redis or a database) tied to that ID, and sends the ID back to the client.

2. JSON Web Tokens (JWTs) (Stateless Architecture)

As the web scaled to globally distributed microservices, maintaining session data in central databases for millions of users became a bottleneck. JSON Web Tokens (JWTs) were introduced as a stateless alternative.

A JWT is a self-contained token containing user claims (like their ID and role) and a cryptographic signature. Because the server can verify the token's authenticity using its secret key without doing a database lookup, it offloads state from the server.

Anatomy of a JWT

A JWT consists of three Base64 URL-encoded parts separated by dots (.):

Metadata & Algorithm] Payload[2. Payload
User Data / Claims] Signature[3. Signature
Cryptographic Verification] JWT --- Header JWT --- Payload JWT --- Signature
  • Header: Specifies metadata, like the signing algorithm used (e.g., HMAC SHA256).
  • Payload: Contains the claims or data. Common fields include sub (subject/user ID) and iat (issued at).
  • Signature: Ensures the token hasn't been tampered with. If an attacker modifies the payload, the signature verification will fail on the server.

3. Cookies

A Cookie is a mechanism that allows a server to store a small piece of information (like a Session ID or a JWT) directly in the user's browser. The true power of cookies lies in automation: once a cookie is set by a server, the browser will automatically attach that cookie to every subsequent HTTP request sent back to that specific server.

Mental Model

Think of Sessions like a coat check at a club. You give the server your coat (data), and it gives you a ticket number (Session ID) to hold.

Think of a JWT like a driver's license. All your information is printed directly on the card (Payload) and it has an official holographic seal (Signature) proving it's authentic.

Important Terms

  • Stateful: Remembering previous interactions (e.g., relying on a server database).
  • Stateless: Treating every request independently; relying on self-contained tokens.
  • Claims: The pieces of information asserted about a subject inside a JWT (like user ID or role).

Key Takeaways

  • Sessions require database or Redis lookups, causing scalability bottlenecks in distributed systems.
  • JWTs scale infinitely because they require zero server-side memory, but they are difficult to revoke once issued.

Quick Revision

  • HTTP is Stateless: Memory had to be artificially added.
  • Sessions: Server stores data, client holds a reference ID.
  • JWTs: Client holds the actual data securely via a signature.
  • Cookies: Browser storage mechanism that automatically sends data back to the issuing server.
