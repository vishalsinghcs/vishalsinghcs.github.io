Interview Questions

13 - Error Handling: Fault Tolerance, Global Middleware, and Security.

These questions evaluate your deep understanding of how to build resilient systems, manage unpredicted failures, and architect secure error propagation. The goal is engineering thinking, not recitation of terms.

1. Scenario-based Questions

  • Your backend application is successfully returning 200 OK for all health checks, but users are complaining that payments are failing. How would you investigate the disconnect between the health checks and the user experience?
  • A new deployment goes out, and suddenly your API starts returning 500 errors across all routes that interact with an external AI service. The external service's status page shows all systems operational. What could be the issue?
  • During a flash sale, your application starts getting thousands of 429 Too Many Requests responses from your transactional email provider. How do you handle this gracefully without losing emails or crashing your system?
  • Users report that their newly registered accounts are occasionally showing negative wallet balances. The application logs don't show any crashes or exceptions. How do you track down this error?
  • Your error monitoring dashboard shows a sudden spike in 404 errors for an endpoint fetching user profiles. However, your database team confirms all requested users actually exist. What could be causing this?
  • You notice that a specific database query occasionally times out during peak hours. You suspect a deadlock. How do you confirm this, and how can the backend application recover from it?
  • Your platform relies on a 3rd party authentication provider which goes down entirely for 2 hours. What proactive strategies could your backend have in place to provide a degraded but functional experience?
  • An attacker is systematically submitting incorrect passwords for thousands of emails. Your monitoring flags this, but the attacker is rotating IPs. How should your login error responses be structured to prevent them from enumerating which emails belong to valid accounts?

2. "What if?" Questions

  • What if you designed a system where every single layer (Controller, Service, Repository) independently caught errors and sent HTTP responses directly to the client? What architectural problems would arise?
  • What if your application blindly logged the full error objects (including stack traces) from the database whenever an exception occurred? What are the security and compliance risks?
  • What if you decided to implement aggressive retry logic (e.g., retrying immediately up to 10 times without delay) for all failed external API calls? What could go wrong?
  • What if your application only validated configuration variables (like API keys) lazily when they were first used by a user request, instead of at startup?

3. Debugging Questions

  • A user attempts to create a new book but receives a raw PostgreSQL constraint violation error message in their browser. Where did the error handling strategy break down in the backend architecture?
  • You have a global error handling middleware, but a specific API route is causing the Node.js/Python process to crash completely, bypassing the middleware. What type of error could bypass a centralized exception handler?
  • Your logs in Datadog show a spike in database connection errors. The database server CPU is low, and network latency is normal. Where else in the backend architecture would you look for the bottleneck?

4. Design Questions

  • Design a resilient webhook delivery system that must send notifications to hundreds of external client servers. How do you handle clients that are temporarily down or rate-limiting your requests?
  • Design a centralized error handling middleware for a REST API. How do you map low-level database errors (e.g., foreign key violations, unique constraints) to standard HTTP status codes without leaking database details?
  • Design a proactive health checking system for a monolithic e-commerce application that relies on Postgres, Redis, Stripe, and AWS S3. What exactly do you test beyond a simple HTTP ping?

5. Architecture & Trade-off Questions

  • When handling failed operations, what are the trade-offs between implementing an Exponential Backoff retry strategy versus failing fast and pushing the error to a Dead Letter Queue for manual intervention?
  • Some teams prefer throwing exceptions (e.g., Python/Java/JS) for business logic errors, while others prefer returning error objects/values (e.g., Go/Rust). What are the architectural trade-offs of treating errors as exceptions versus treating them as values?
