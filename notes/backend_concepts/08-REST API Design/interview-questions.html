Interview Questions

08 - REST API Design: Resources, URLs, HTTP Methods, CRUD Patterns, Pagination, Custom Actions, and Best Practices

These questions test your ability to design intuitive, consistent, and scalable REST APIs — reasoning about URL structure, status codes, idempotency, pagination strategies, and trade-offs between PATCH and PUT. The goal is engineering thinking, not memorization.

1. Scenario-based Questions (40%)

  • Your team has built an API for a SaaS platform with 30 endpoints. A new frontend engineer joins and immediately asks why the user creation endpoint uses description but the project creation endpoint uses desc for the same concept. She also notices that some list endpoints return a 404 when no results match a filter, while others return an empty array with 200. How did the codebase get into this state, and what is the process and technical debt cost of fixing it in a live system with production clients?
  • You are designing a REST API for a subscription billing system. When a user's trial period ends, the system needs to: lock their account, cancel all active scheduled jobs, archive their data, send a billing email, and log a compliance event. A junior engineer proposes handling this by having the client call PATCH /users/:id with {"status": "trial_expired"}. What is wrong with this approach and how would you design the API endpoint instead?
  • Your mobile app is calling GET /notifications without any pagination parameters. In development (100 records) everything works fine. After six months in production with 500,000 users, this endpoint is timing out and taking down the API server. Walk through the exact changes you would make — at the API interface level and at the server implementation level — to fix this without breaking existing mobile clients that cannot be force-updated immediately.
  • A third-party integrator is building against your public API. They report that after calling POST /invoices and receiving a network timeout (no response received), they are now unsure whether the invoice was actually created in your system or not. They are afraid to retry because they don't want to create a duplicate invoice. How does the concept of idempotency apply here, and how would you redesign the endpoint or the client retry strategy to make this safe?
  • Your API has a GET /products list endpoint. A customer wants to filter by both category=electronics and priceRange=100-500 simultaneously, sort by rating in descending order, and get the second page of 20 results. Write out the exact URL the client should send. Then explain what the server's paginated response object should look like and what each field in it means.
  • You are designing a REST API for a healthcare platform. A HIPAA compliance requirement states that all data access must be logged with the reason for access. A clinical system calls GET /patients/:id/records to fetch a patient's records. How do you pass the reason for access in a GET request without violating REST conventions (no body in GET), and without polluting the URL with sensitive data that might appear in server access logs?
  • Your REST API has been deployed to production for 18 months. You need to introduce a breaking change — an existing field phone (which returns an unformatted string like "9876543210") needs to be replaced with a structured phoneNumber object ({"countryCode": "+91", "number": "9876543210"}). Hundreds of clients depend on the old format. How do you handle this transition at the API design level without breaking existing integrations?
  • Your team is building a multi-tenant SaaS platform. An API endpoint GET /projects should return only the projects belonging to the currently authenticated user's organization — not all projects in the database. A junior engineer implements this by requiring the client to pass ?organizationId=abc123 as a query parameter. What is the security and design problem with this approach, and how should it be implemented correctly?

2. "What if?" Questions (20%)

  • What if a REST API had no versioning at all (/organizations instead of /v1/organizations)? The company has 50 active third-party integrators. The backend team needs to make a breaking change to the POST /organizations response shape. Walk through every consequence this causes and how you would resolve the situation retroactively.
  • What if a list API returned all resources with no pagination, no filtering, and no sorting? The system starts with 100 records and everything seems fine. Trace the exact sequence of events over 2 years — as records grow to 1 million — showing where performance degrades, what breaks first, and what the cascading effects are on the rest of the infrastructure.
  • What if every operation in your REST API — including fetches, updates, and deletes — used only POST, because the original developer thought it was "simpler"? A new team member joins and needs to integrate the API. What specific problems do they encounter, and how does violating the HTTP method semantics affect tooling, caching layers, and security middleware?
  • What if a DELETE endpoint returned the deleted object in the response body with a 200 status code instead of 204 with an empty body? Some integrators begin relying on this behavior to avoid an extra GET call. Six months later you want to fix it to the standard 204. What has your earlier decision cost you and how do you fix it?

3. Debugging Questions (15%)

  • A client is calling GET /orders?status=pending and sometimes receiving a 404, but sometimes receiving a 200 with an empty array — from the same codebase, depending on which server instance handles the request. No database changes are happening between calls. What are the three most likely root causes of this inconsistency and how would you confirm each one?
  • An iOS client calls PATCH /users/me to update a user's profile photo URL. The endpoint works correctly in Postman but returns 400 on the mobile app. The request body is identical. What are the most likely differences between the Postman and mobile app requests that could cause this, and how do you isolate the issue?
  • Your GET /reports list endpoint is returning the correct data for small datasets but is skipping records when paginated with page=2&limit=10. Users report seeing the same records on page 1 and page 2, and some records never appear. You haven't changed the query logic. What pagination implementation bug is most likely responsible for this and how do you fix it?

4. Design Questions (15%)

  • Design the complete REST API surface for a simplified e-commerce platform with three resources: products, orders, and reviews. For each resource, list all endpoints (method + URL), the request body fields (excluding server-managed fields), the success response codes, and any custom actions you would add. Include the pagination, sorting, and filtering parameters you would support on each list endpoint.
  • Design an API for a real-time collaborative document editor (similar to Google Docs) where multiple users edit simultaneously. The system needs endpoints for creating documents, fetching document content, saving changes, sharing with collaborators, and managing permissions. Which operations would you model as custom actions rather than standard CRUD and why? What status codes apply to each?
  • Design a public REST API for a weather service that serves 10 million requests per day. The API must support: current weather by city, 7-day forecast by coordinates, historical data by date range, and a webhook subscription for real-time alerts. For each of these, design the URL structure, the query parameters, the response shape, and any caching strategy you would embed in the API contract itself.

5. Architecture & Trade-off Questions (10%)

  • REST APIs are "stateless" by constraint — the server stores no session information between requests. Some applications work around this by embedding state in JWTs or by using sticky sessions at the load balancer. Explain the exact trade-offs between true statelessness and these workarounds in terms of: horizontal scalability, session revocation, and the complexity of your infrastructure.
  • A product manager asks you to add a bulk delete endpoint — the client should be able to send a list of up to 1,000 IDs and delete all of them in one call. The REST specification doesn't have a clean standard for this. Discuss at least three different API designs you could use to implement this, with the trade-offs of each in terms of RESTfulness, safety, and client complexity.
