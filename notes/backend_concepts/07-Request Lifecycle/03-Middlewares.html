Middlewares

Functions that execute in the middle of the request lifecycle — between the entry point, routing, handlers, and the final response — to handle cross-cutting concerns without code duplication.

What is Middleware?

From the entry point of a server to the moment a response is sent back to the client, the request passes through multiple function boundaries. Functions placed at these boundaries are called middlewares.

Like regular handlers, a middleware receives the request and response objects from the runtime — but it also receives a third thing: a next function.

  • next(): Calling this function passes execution to the next middleware in the chain. Without calling next(), the request stops at the current middleware.
  • A middleware can also choose to terminate the request by sending a response directly (e.g., returning a 401 Unauthorized from an auth middleware), preventing execution from ever reaching the handler.

Why Middlewares Exist

The same reason we use functions in programming — to avoid repeating the same code in multiple places. A backend application may have hundreds of API endpoints. Without middlewares, common logic like authentication, logging, or CORS checks would need to be duplicated in every single handler. Middlewares allow this shared logic to run once per request, automatically, without polluting the handlers.

The Middleware Chain

flowchart LR Client([Client Request]) --> MW1[CORS Middleware] MW1 -->|next| MW2[Logging Middleware] MW2 -->|next| MW3[Auth Middleware] MW3 -->|next| Router[Route Matching] Router --> MW4[Rate Limit Middleware] MW4 -->|next| Handler[Controller / Handler] Handler --> MW5[Global Error Handler] MW5 --> Response([Response to Client]) MW3 -- "401 Fail" --> Response MW4 -- "429 Fail" --> Response

Ordering Matters

The order in which middlewares are registered is the order in which they execute. This is critical. The typical production-grade ordering is:

  1. CORS — Terminate unauthorized origins at the earliest possible point.
  2. Logging / Monitoring — Record each request before it proceeds.
  3. Authentication — Verify identity; reject unauthenticated requests early.
  4. Rate Limiting — Protect the server from abuse.
  5. Handler Logic — The actual controller/service/repository stack.
  6. Global Error Handler — Must be last so it can catch errors from all previous steps.

Common Middleware Examples

CORS

Browsers enforce a Same-Origin Policy that blocks web apps from accessing resources on a different domain. The CORS middleware checks the Origin header of the incoming request. If the origin is on the allowed list, it adds Access-Control-Allow-Origin headers to the response and calls next(). If not, it does nothing and the browser blocks it by default.

Authentication

Extracts a token (JWT, session ID) from the request headers. Verifies it. If invalid → returns 401 Unauthorized immediately. If valid → extracts user info (user ID, role) and stores it in the request context (covered in the next note), then calls next().

Rate Limiting

Tracks how many requests a given IP address has made within a time window. If the client exceeds the threshold → returns 429 Too Many Requests. If within limits → calls next().

Global Error Handling

Placed last in the chain. Catches any unhandled error thrown anywhere in the entire request lifecycle (from any handler, service, or other middleware). Converts it into a properly structured JSON error response (400 for client errors, 500 for server errors) and sends it back to the client. Placing this anywhere other than last means it misses errors from subsequent steps.

Logging

Runs early in the chain to record information about every incoming request: the path, method, query parameters, and request body. This data is invaluable for debugging and auditing.

Mental Model

Think of middlewares as airport security checkpoints. Every passenger (request) must pass through identity check (auth), baggage scan (validation), boarding pass check (rate limit) before reaching the gate (handler). If any checkpoint fails, the passenger is turned away early — they never reach the gate.

Important Terms

  • Middleware: A function that executes between two stages of the request lifecycle, with access to req, res, and next.
  • next(): A function that passes control to the next middleware in the chain.
  • 429: HTTP status for "Too Many Requests" — returned by rate limiting middleware.
  • Cross-cutting concern: A feature needed by many parts of the app (auth, logging) — middleware's ideal use case.

Key Takeaways

  • Middleware = shared code that runs for every (or many) requests, without duplication.
  • A middleware can either call next() to continue or send a response to terminate the request.
  • Order is critical — CORS first, Global Error Handler last.
  • Authentication middleware stores user info in request context, not in global state.

Quick Revision

  • Middleware receives: req, res, next.
  • CORS: Allow/block by origin.
  • Auth: Verify token → 401 or next().
  • Rate Limit: Too many requests → 429.
  • Error Handler: Always last in chain.
  • Logging: Always early in chain.
