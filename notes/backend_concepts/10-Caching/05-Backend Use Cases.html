Backend Use Cases for Caching

In daily backend development, when do you actually write code to interact with Redis or Memcached? Here are the four most common real-world scenarios.

1. Database Query Caching

Some SQL queries are inherently expensive. They might join multiple large tables and perform complex aggregations. If an endpoint calling this query is hit frequently (e.g., loading a dashboard), it will overload the database.

  • Solution: Run the query once, cache the JSON result in Redis with a TTL of 1 hour.
  • E-commerce Example: Amazon caches static product details (like a Macbook description) during a sale. Hitting the database a million times a minute for static data is wasteful; serving it from RAM protects the database so it can handle actual purchases.
  • Social Media Example: Twitter profiles are heavily read but rarely updated. Caching a celebrity's profile prevents the DB from being overwhelmed when millions view it in a single day.

2. Session Storing (Authentication)

In a stateful authentication flow, a server generates a session token upon login. For every subsequent API request, the server must validate that token.

  • The Problem: Querying Postgres to validate a session token on every single API call adds massive latency and unnecessary database load.
  • The Solution: Store active session tokens in Redis. Because fetching from RAM is instantaneous, API calls remain fast and the primary database is spared.

3. API Caching (3rd Party Integrations)

Often, your backend needs to talk to an external API (e.g., a Weather API) to serve data to your frontend.

  • The Problem: External APIs usually have strict rate limits and charge you based on usage. If 1,000 users load your app, you make 1,000 calls to the Weather API, risking blocks or high bills.
  • The Solution: Fetch the weather data once and cache it in Redis with a TTL of 30 minutes. The next 999 users are served the cached weather data. You satisfy 1,000 users while only paying for 1 API call.

4. Rate Limiting

Rate limiting prevents bots or malicious users from crashing your server by limiting how many requests they can make (e.g., "Max 50 requests per minute").

How it works with Redis:

  1. A middleware intercepts the request before it reaches the controller.
  2. It reads the user's public IP address from the X-Forwarded-For header.
  3. It uses the IP as a Key in Redis, and stores a counter as the Value.
  4. Every request increments the counter. If the counter exceeds 50 within a 1-minute TTL window, the middleware returns a 429 Too Many Requests error.

Why Redis? Tracking rate limits requires an update on literally every single HTTP request. Doing this in a relational database would flood it with write operations. Redis handles this effortlessly due to its in-memory speed.

Mental Model

Think of API Caching like subscribing to a newspaper. Instead of calling the journalist (External API) every 5 minutes to ask if the news changed, you buy the paper once (Cache) and share it with your family for the rest of the day.

Important Terms

  • Rate Limiting: Restricting the number of network requests a user can make in a given timeframe.
  • X-Forwarded-For: An HTTP header that identifies the originating IP address of a client connecting to a web server through a proxy or load balancer.
  • Stateful Auth: Authentication where the server must explicitly remember and track active sessions.

Key Takeaways

  • Never query static or slow-changing data from a DB on heavy-traffic endpoints.
  • Always cache 3rd party API responses to save money and avoid rate limits.
  • Use Redis for high-frequency writes that don't need permanent guarantees, like tracking rate limit counters.
