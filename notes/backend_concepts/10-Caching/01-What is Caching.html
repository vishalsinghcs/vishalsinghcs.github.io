What is Caching?

Caching is a fundamental mechanism used to decrease the time and effort it takes to perform computation or retrieve data. By storing a subset of data in a faster, more accessible location, we drastically improve application performance.

The Core Definition

In one sentence: Caching decreases the time and effort to perform work.

More technically, caching involves keeping a subset of primary data in a location that is much faster to access. The decision of what to cache depends on access frequency, usage probability, and the computational cost of generating that data.

We only cache a subset of data because fast storage (like RAM) is expensive and limited in capacity compared to primary storage (like hard disks). We cannot cache everything—we must selectively cache data to optimize performance.

Real-World Examples of Caching

Caching is the secret engine behind almost every high-performance, globally scaled platform.

1. Google Search (Distributed In-Memory Caching)

When you search "what is the weather today" on Google, the search engine doesn't crawl, index, and rank billions of web pages from scratch. Doing so would require immense CPU and memory resources and result in high latency.

  • Google uses a distributed in-memory caching system spread across the globe.
  • If the query result is already stored in the cache (a cache hit), it returns instantly.
  • If the query result is not found (a cache miss), the engine computes the result, returns it, and then stores it in the cache for future users.

2. Netflix (Content Delivery Networks)

Netflix streams terabytes of video content to millions of users worldwide with minimal buffering. They achieve this using CDNs (Content Delivery Networks).

  • Instead of serving all videos from originating servers in the US, Netflix places Edge Servers geographically closer to users.
  • Algorithms predict which content is popular in specific regions and cache that subset of content on the edge servers.
  • Users in India stream from servers in India, drastically reducing latency and preventing the originating servers from crashing under global load.

3. Twitter / X (Trending Topics)

Twitter calculates "Trending Topics" by analyzing billions of tweets in real-time. This machine learning computation is highly resource-intensive.

  • Running this computation every time a user opens the trending page would crash the servers instantly.
  • Instead, Twitter computes the trends periodically (e.g., every few minutes) and stores the results in an in-memory cache like Redis.
  • When you open the app, you are served the pre-computed cached result, leading to instantaneous load times.

When Should We Use Caching?

From the examples above, a clear pattern emerges. You should implement caching when:

  1. You want to avoid repeating heavy computations.
  2. You want to avoid repeatedly transferring large amounts of data over long distances.
  3. The data being requested is frequently accessed and does not change every millisecond.

Mental Model

Think of caching like a wallet vs. a bank vault. The bank vault (database) stores all your money and is very secure, but it takes a long time to travel there and retrieve it. Your wallet (cache) stores a small subset of that money. It's much faster to pull cash from your wallet when buying coffee, saving you the time and effort of going to the bank every time you need to make a purchase.

Important Terms

  • Cache Hit: The requested data was found in the cache and returned immediately.
  • Cache Miss: The data was not in the cache, requiring the system to compute it or fetch it from primary storage.
  • Subset: Caching never stores 100% of the data; it selectively stores what is most useful.
  • Latency: The time delay before data transfer begins. Caching aims to minimize this.

Key Takeaways

  • Caching decreases both time (latency) and effort (server load/compute).
  • It relies on storing a carefully chosen subset of data in faster storage mediums.
  • Major platforms (Google, Netflix, Twitter) rely entirely on caching to survive global scale.
  • Use caching for compute-heavy tasks or massive data transfers that don't change dynamically every second.

Quick Revision

  • Google: Caches search results to avoid re-running expensive ranking algorithms.
  • Netflix: Caches massive video files on edge nodes closer to users.
  • Twitter: Caches compute-heavy trending topic data.
  • Rule of thumb: Cache when computation is heavy, data is large, and changes are infrequent.
