Request Context

A shared, request-scoped state that allows middlewares and handlers to communicate data without being directly coupled to each other.

The Problem It Solves

In the middleware chain, multiple functions execute sequentially for a single request. Consider the authentication middleware — it verifies the user's token and discovers the user's ID and role. Now the handler needs that user ID to insert a record correctly. How does the handler get it?

  • Wrong approach: Pass the user ID directly as a function parameter — this tightly couples the middleware and handler together.
  • Wrong approach: Store it in a global variable — this causes race conditions when thousands of requests run concurrently.
  • Correct approach: Store it in the request context — a shared state scoped to this single request only.

What is Request Context?

Request context is a key-value store that is:

  • Scoped to one request: Created when a request arrives and destroyed when the response is sent. It is completely isolated from other concurrent requests.
  • Accessible to all: Every middleware and handler in the chain for that request can read from and write to the context.
  • Framework-provided: Every major backend runtime provides its own implementation — context.Context in Go, the req object's custom properties in Express, request.state in Python frameworks, etc.

Core Use Cases

1. Passing Authentication Data

The auth middleware verifies a JWT, extracts userId and userRole, and stores them in the context. Any downstream handler then reads from context instead of trusting user-supplied data in the request body — which would be a security vulnerability (a malicious client could fake any user ID).

2. Request ID / Tracing

An early middleware generates a unique ID (UUID) for the request and stores it in context. Every log line, every downstream service call, and every error message uses that same ID. This makes it possible to trace a single request's journey across dozens of microservices in a distributed system.

3. Cancellation and Deadlines

In Go (and similar languages), the context is also used to carry cancellation signals and deadlines. If a client disconnects before the server responds, the context is cancelled. Any downstream operation (database query, external API call) that respects the context will abort immediately, freeing up resources instead of hanging indefinitely.

Security Implication

Never use user-provided data from the request body to identify who the user is. Always read identity from the request context (where it was placed by the authentication middleware after token verification). This is a critical security boundary:

Source Trusted? Why
Request body userId field ❌ Never Any client can send any value — easily spoofed
Request context userId ✅ Yes Set by auth middleware after cryptographic token verification

Mental Model

Think of request context as a boarding pass folder for each airline passenger (request). The check-in agent (auth middleware) puts the passenger's seat number and meal preference (userId, role) inside. Every flight attendant (subsequent handler/middleware) can open that folder and read the info — without the passenger having to repeat it every time. And critically, one passenger's folder never gets mixed up with another's.

Important Terms

  • Request Context: A request-scoped key-value store accessible by all middlewares and handlers for that single request.
  • Request Scoping: Data is isolated per request — concurrent requests cannot see each other's context.
  • Cancellation Signal: A signal propagated through the context to abort downstream operations when a request is cancelled.
  • Tracing / Request ID: A unique ID stored in context to track a request across services in logs.

Key Takeaways

  • Request context decouples middlewares from handlers — neither needs to know about the other directly.
  • Always read user identity (userId, role) from context, never from the request body.
  • Context is request-scoped — it is not a global store. Concurrent requests have separate contexts.
  • Context can carry deadlines and cancellation signals to prevent server hangs on slow downstream calls.

Quick Revision

  • What: Key-value store scoped to one HTTP request.
  • Who sets it: Middlewares (e.g., auth middleware).
  • Who reads it: Subsequent middlewares and handlers.
  • Use case 1: Pass userId/role from auth to handler.
  • Use case 2: Attach request ID for distributed tracing.
  • Use case 3: Propagate cancellation/deadline signals.
